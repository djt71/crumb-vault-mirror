# Agent-to-Agent Communication — Action Plan

**Source:** `design/specification.md` (25 tasks, 4 phases, reviewed)
**Scope:** Full PLAN for Phase 1 + Phase 1b (actionable). Phase 2-4 sketched at milestone level.

## Key Decisions (resolving open questions)

### OQ1: Orchestration Logic Hosting → Discrete Skills per Workflow

Each workflow gets its own OpenClaw cron job or skill invocation. Shared infrastructure (delivery, feedback, context) lives in SOUL.md instructions and `_openclaw/state/` schemas. No unified orchestration engine.

**Rationale:** Matches existing operational pattern (awareness-check is a bash script + cron, morning briefing is a prompt-based skill). Avoids monolith risk. Each workflow has its own cadence, trigger, and failure domain. Shared state is in vault files (blackboard pattern), not in-memory.

### D1: Terminology Glossary

| Term | Definition | Scope |
|------|-----------|-------|
| `correlation_id` | UUID linking a full orchestration chain: trigger → decision → dispatch → feedback → learning. Generated by Tess at decision time. | Cross-cutting |
| `dispatch_id` | CTB-016 dispatch lifecycle identifier. Generated by the bridge runner. One per Crumb session. | Bridge transport |
| `workflow` | Named orchestration path (e.g., `compound-insight`, `research`, `account-prep`). | Tess routing |
| `intent` | Delivery channel action type: `notify`, `present`, `approve`, `feedback`, `converse`. | Delivery layer |
| `capability_id` | `domain.purpose.variant` identifier for skill dispatch. | Capability resolution |
| `dispatch_group_id` | Deferred to Phase 4 if multi-dispatch (§8.5) is warranted. Would link sequential dispatches within a single orchestration chain. | Future |

## M1: Foundation Infrastructure

**Spec tasks:** A2A-001, A2A-002, A2A-003
**Dependencies:** TOP-009 operational ✓, FIF M3 done ✓
**Estimated effort:** ~2-3 sessions
**Success criteria:** Tess can deliver through abstract interface, context model is populated and refreshable, feedback ledger accepts signals with correlation IDs.
**Exit stability thresholds:** (1) Delivery: at least one non-A2A test workflow (e.g., morning briefing) using `deliver(intent, …)` without errors for 2 days. (2) Context: morning briefing has successfully refreshed `tess-context.md` 3+ times without exceeding size limit. (3) Feedback: ledger has at least 3 real entries; schema has not changed between first and third entry.

### Implementation approach

All three tasks are infrastructure that Workflow 1 consumes. They should be built and smoke-tested before Workflow 1 integration.

**A2A-001 (Delivery layer):** Define the `deliver(intent, content, artifact_path?)` contract in SOUL.md. Implement as Tess-side instructions — she calls existing OpenClaw Telegram delivery but through the abstract intent vocabulary. Delivery envelope schema defined in `_openclaw/state/delivery-config.yaml`. No new code repo — this is prompt engineering + schema definition.

**A2A-002 (Context model):** Create `_openclaw/state/tess-context.md` with defined schema. Amend morning briefing (TOP-009) to refresh it. Add `refreshed_at` timestamp. Staleness tiers are instruction-based (SOUL.md).

**A2A-003 (Feedback infra):** Define `feedback-ledger.yaml` schema. Add SOUL.md instructions for Tess to record feedback from Telegram reactions. Correlation ID generation in SOUL.md. Mechanical coupling to learning log is instruction-based (deferred to A2A-011 implementation). Feedback timeout mechanism: 24h timeout per workflow, pending items tracked in ledger with `awaiting_feedback` status, `no-feedback` outcome signal recorded on timeout, no reminder nudge (silent close).

## M2: Compound Insight Workflow

**Spec tasks:** A2A-004, A2A-005
**Dependencies:** M1 complete (exit stability thresholds met), FIF M3 done ✓
**Estimated effort:** ~3-4 sessions (including iterations)
**Success criteria:** End-to-end compound insights flowing: feed trigger → Tess cross-reference → Crumb dispatch → vault write → delivery → feedback. 3-day gate passed.

### Implementation approach

A2A-004 is the largest Phase 1 task. Split into three implementation tasks:

**A2A-004.1 (Schema + dispatch template):** Define compound insight frontmatter schema. Create Crumb dispatch template (the bridge dispatch request that tells Crumb what to write). Define dedup rules (exact source_item + 7-day temporal).

**A2A-004.2 (Orchestration trigger + cross-reference):** Implement the cron job or skill that Tess runs to process feed-intel items. SOUL.md instructions for tier-based selection (all T1 + T2 matching project tags), cross-reference against tess-context.md projects, noise ceiling enforcement (3/day during gate, 5/day after).

**A2A-004.3 (End-to-end integration + smoke test):** Wire everything together. Budget 4-6 live deployment iterations per Pattern 4. First real compound insight through the full pipeline. Fix what breaks.

**A2A-004.2 specifics:** Disable threshold for patterns: window = last 20 items or 7 days (whichever is larger), minimum N = 10 items, scope = per-pattern key (not global).

**A2A-005 (Gate evaluation):** 3-day observation after A2A-004.3 succeeds. Run Haiku vs Sonnet A/B comparison (5 items each) as first activity. Document gate decision. **SOUL.md drift gate question (added per review):** "Did any SOUL.md-instructed deterministic operations (envelope format, ledger schema, correlation ID format) produce inconsistent outputs that required manual correction?" If yes → build targeted bash helpers for those specific operations before M3. If no → proceed.

## M3: Capability Infrastructure

**Spec tasks:** A2A-006, A2A-006.5, A2A-007, A2A-007.5, A2A-008, A2A-009
**Dependencies:** A2A-006/007/007.5 can start after M1 is built (no M2 gate dependency — schema work has zero logical dependency on W1 outcomes). A2A-008/009 gated on M2 gate passed (A2A-005) — resolution and quality gate logic benefits from W1 operational experience.
**Estimated effort:** ~2-3 sessions
**Success criteria:** Capability manifest schema defined, all manifests validate, existing skills + vault-query have manifests, Tess can resolve `research.external.*` → researcher-skill, adaptive quality gates filter by manifest.
**Test harness:** Given two skills with different `cost_profile` and `supported_rigor`, a test brief with `rigor=standard` is routed to the correct one; log shows chosen skill and rationale.

### Implementation approach

This milestone is schema-first: define the contracts, validate, add manifests, then build resolution logic.

**A2A-006 (Manifest schema + brief schema):** Define `_system/schemas/capabilities/manifest.yaml`. Extract `_system/schemas/briefs/research-brief.yaml` from researcher-skill's current brief structure. Document ID namespace convention and substitution test.

**A2A-006.5 (Manifest validation script):** Create `_system/scripts/manifest-check.sh` — loads all SKILL.md files with `capabilities` frontmatter, validates against manifest schema, fails hard on mismatch. Dependency for A2A-008 (resolution logic reads validated manifests, not raw frontmatter).

**A2A-007 (Skill manifests):** Add `capabilities` frontmatter block to researcher-skill SKILL.md and feed-pipeline SKILL.md. 2 files changed. Run manifest-check.sh to validate.

**A2A-007.5 (Vault-query skill):** Create `.claude/skills/vault-query/SKILL.md` declaring `vault.query.facts` capability. Accepts `vault-query-brief.yaml` schema (account/topic, output format, scope constraints). Uses obsidian-cli internally for vault searches. Produces structured output (relevant notes, recent activity, open items). Single session. This is what Workflow 3's vault query branch dispatches to.

**A2A-008 (Capability resolution):** SOUL.md instructions for Tess's resolution algorithm. Pre-computed `_openclaw/state/capabilities.json` index built from SKILL.md manifests (reduces runtime parsing complexity for Haiku). Filter by `supported_rigor` compatibility. Rank by learning log → cost → quality signals → model tier → alphabetical tiebreak. Dispatch request format amendment. **Cold-start restriction:** capabilities with <3 learning log entries cannot be auto-selected for `rigor: deep` briefs — escalate to operator. **Initial implementation uses manifest `cost_profile` as ranking fallback** until learning log (A2A-011) is populated.

**A2A-009 (Quality review schema):** SOUL.md instructions for adaptive quality gates. Validate researcher-skill output format against gate expectations. Define auto-deliver / re-dispatch / escalate decision tree. Hard limits (max 2 re-dispatches) stored in dispatch state, not just instructed.

## M6: SE Account Prep (Phase 2)

**Spec tasks:** A2A-016, A2A-017, A2A-018
**Dependencies:** M3 capability resolution ✓, M4 research pipeline ✓, TOP-027 calendar ✓, customer-intelligence dossiers (ACT — 3 live dossiers)
**Estimated effort:** ~3-4 sessions
**Success criteria:** Automated pre-call briefs via sequential dispatch (`vault.query.facts` → `research.external.*` → synthesis), deadline-aware scheduling from calendar + learning log, delivered ≥20 min before meetings. 3-day gate passed.
**Exit stability thresholds:** (1) At least 2 real pre-call briefs delivered to Danny before meeting time. (2) Scheduling precondition correctly blocked at least 1 "too late" case or no such cases arose. (3) Sequential dispatch completed without manual intervention in both dispatches. (4) Post-call feedback captured for each brief.

### Implementation approach

A2A-016 is lightweight alignment work. A2A-017 is the large task, split into three implementation tasks. A2A-018 is the gate.

**A2A-016 (Dossier schema alignment):** The customer-intelligence project already established the dossier schema and produced 3 live dossiers (ACG, BorgWarner, Steelcase). This task bridges the existing schema with A2A queryability requirements:
- Add `engagement_state` field to dossier template and live dossiers (values: `active`, `at-risk`, `dormant`, `new`). Currently implicit from `health` field — make explicit for machine routing.
- Verify `last_refreshed` field is queryable as the staleness signal (spec says `last_touch_date` — align naming with existing `last_refreshed` field rather than introducing a second timestamp).
- Add `contacts` list to dossier frontmatter (name, role, last_interaction). Currently in prose body — surface to frontmatter for Tess querying.
- Validate that `vault.query.facts` capability (A2A-007.5) can extract structured data from dossier frontmatter.
- Update dossier-template.md and backfill live dossiers.

**A2A-017.1 (Orchestration template + SOUL.md):** Define the W3 orchestration flow in SOUL.md:
- Calendar trigger: Tess checks upcoming meetings (from morning briefing calendar section, TOP-027) and identifies accounts needing prep.
- Scheduling precondition: compute expected completion time from `dispatch-learning.yaml` averages for `vault.query.facts` and `research.external.*` capabilities. If `now + expected_duration + 20min_buffer > meeting_time` → escalate "too late to fully prep" with partial brief option.
- Context freshness check: if `tess-context.md` is >6h stale, lightweight refresh (project-state.yaml files only) before brief formulation.
- Create dispatch template for W3 at `_openclaw/dispatch/templates/account-prep.md`.
- Define synthesis prompt structure: vault facts as primary, research as supplement, surface contradictions between internal knowledge and external findings.

**A2A-017.2 (Sequential dispatch implementation):** Wire the two-stage dispatch:
- Stage 1: Formulate `vault.query.facts` brief (account name, scope: dossier + related notes + recent interactions). Resolve capability → dispatch to Crumb → collect vault query results.
- Stage 2: Formulate `research.external.*` brief informed by vault query gaps (what's stale, what's missing, what contradicts public information). Resolve capability → dispatch → collect research results.
- Tess holds Stage 1 output as working state while Stage 2 executes (intermediate results don't count against situational context budget per updated §8.1).
- Error handling: Stage 1 failure → escalate (can't prep without vault context). Stage 2 failure → deliver vault-only brief with flag "external research unavailable."

**A2A-017.3 (Synthesis + delivery + feedback):** Wire the final pipeline stage:
- Tess synthesizes pre-call brief from both dispatch results. Brief structure: account snapshot (from dossier), recent developments (from research), talking points, open items, risk flags.
- Deadline-aware delivery: if meeting < 30 min away at synthesis time, deliver partial with "⚠ Incomplete — research still pending" flag.
- Delivery via `present` intent with artifact path to the vault brief.
- Post-call feedback prompt via `feedback` intent: "How was the prep? React 👍/👎 or reply with notes."
- Feedback recorded in ledger with `workflow: account-prep` and correlation_id linking to both dispatches.
- Budget 3-4 live deployment iterations per Pattern 4.

**A2A-018 (Gate evaluation):** 3-day observation after A2A-017.3 succeeds. Gate criteria:
- Timeliness: briefs delivered ≥20 min before all meetings during observation
- Completeness: both dispatch stages completed for each brief (or graceful degradation documented)
- Feedback: ≥1 feedback signal received per brief
- Cost: per-brief cost within acceptable range (establish baseline during gate)
- Gate decision documented in `_openclaw/state/gates/a2a-018-gate-YYYY-MM-DD.md`

## M7: Approval Integration (Phase 2)

**Spec tasks:** A2A-019
**Dependencies:** TOP-049 approval contract ✓, mission-control Phase 2 complete ✓
**Estimated effort:** ~2 sessions
**Success criteria:** AID-* approval requests flow through the delivery abstraction to any configured channel. Multi-channel resolution with first-response-wins semantics. Mission control approval UI coordinated as cross-project work with MC.
**Exit stability thresholds:** (1) At least 2 real AID-* approvals processed through delivery layer. (2) TTL expiry handled gracefully (escalation, not silent failure). (3) Idempotency key prevents duplicate approvals from multiple channels.

### Implementation approach

A2A-019 is rescoped from the original sketch. The mission-control project owns the web UI (MC Phase 3, M8-M10). This milestone focuses on the A2A-side protocol wiring.

**A2A-019.1 (Approval delivery adapter):** Wire the `approve` intent into the delivery abstraction with full approval contract support:
- AID-* token generation: Tess generates a unique approval ID when requesting HITL decisions.
- Approval envelope: `{aid, action_summary, options[], ttl, idempotency_key, requesting_workflow, correlation_id}`.
- Telegram adapter: inline buttons with AID-* reference. Response captured → approval recorded in ledger.
- TTL enforcement: if no response within TTL, escalate (re-deliver with urgency flag, then block workflow if still no response).
- Idempotency: first response wins. Subsequent responses on any channel logged but ignored with "already resolved" reply.
- Channel priority: primary channel (currently Telegram) gets the approval request. Secondary channels (web UI when available) get a notification with link to approve.

**A2A-019.2 (Mission control coordination):** Cross-project task — MC owns implementation, A2A defines the contract:
- Define the approval API contract that mission-control needs to implement: `GET /api/approvals/pending`, `POST /api/approvals/:aid/resolve`.
- Approval state lives in `_openclaw/state/approvals/` (A2A-owned), MC reads and writes through the API.
- Web UI approval panel design: pending approvals list, action summary, approve/deny buttons, resolution history.
- This task produces a coordination document in `design/` — MC implementation is tracked in the mission-control project.

## Risk Summary

| Risk | Mitigation |
|------|-----------|
| Haiku insufficient for Workflow 1 judgment calls | A/B comparison in A2A-005 gate; Sonnet elevation path defined |
| Live deployment iterations exceed budget | Pattern 4 budgets 3-6 per workflow; layered validation reduces iterations. Schema/envelope bugs should be caught in dry-run before counting against live budget. |
| Tess context staleness degrades decisions | Tiered staleness model with mechanical enforcement |
| Feedback cold-start (no learning data) | Capability manifest `cost_profile` as fallback; N-entries heuristic for escalation calibration. Cold-start capabilities (<3 entries) cannot be auto-selected for high-rigor briefs — escalate to operator. |
| Phase 2/3 blocked longer than expected on tess-ops | Phase 1/1b is fully independent; no wasted work |
| SOUL.md prompt drift / untestability | **Deferred code layer:** SOUL.md-as-orchestration is the right starting point (matches existing pattern, no dev overhead). M2 gate (A2A-005) evaluates whether deterministic operations (envelope format, ledger schema, correlation ID format) produce inconsistent outputs. If yes → build targeted bash helpers before M3. If no → proceed. Validation fixtures (folder of example briefs + expected decisions) added at M3 for regression testing after SOUL.md edits. |
| Ledger growth / unbounded state files | Rolling archival windows for feedback-ledger and dispatch-learning (defined at Phase 3). M1/M2 volumes are low enough that growth is not a Phase 1 concern. |
| Schema drift without validation | Manifest validation script (A2A-006.5) validates SKILL.md manifests against schema. vault-check integration for envelope/ledger schema validation at commit time. |
| Correlation ID integrity across systems | correlation_id required in all delivery envelopes, feedback entries, and learning log entries. Cross-reference consistency checked during gate evaluations. |
