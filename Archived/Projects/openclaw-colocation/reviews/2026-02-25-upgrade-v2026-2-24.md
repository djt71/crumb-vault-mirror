# Peer Review: OpenClaw Upgrade Runbook v2026.2.17 to v2026.2.24

**Review prompt:** Structured review of upgrade runbook — evaluating correctness, completeness, internal consistency, feasibility, clarity. Specific focus on rollback safety, HIGH-risk mitigations, config change approach, verification gaps, and launchctl domain correctness.

## OpenAI (gpt-5.2)

## Overall assessment

The runbook is generally **feasible and well-structured**: it sequences stop -> upgrade -> doctor/migrate -> manual deltas -> restart -> isolate/verify -> rollback, and it explicitly calls out the two most likely "it looks fine but it's broken" changes (heartbeat DM drop + exec lockdown). The biggest weaknesses are (a) a couple of **unsafe/fragile ops details** (backup/restore semantics; launchctl domain assumptions; config editing not being atomic), (b) a few **internal inconsistencies/typos** that will confuse an operator, and (c) several **claims that need grounding** (issue numbers and behavior assertions).

## Findings

- **[F1]**
  - **Severity:** CRITICAL
  - **Finding:** Backup/restore is not safe as written: `cp -r` to `.openclaw.bak-*` and restore with `rm -rf ~/.openclaw; cp -r backup ~/.openclaw` can silently produce incorrect nesting/permissions depending on whether the destination exists and how `cp` behaves on macOS (e.g., ending up with `~/.openclaw/.openclaw/...` or losing modes/ownership/ACLs/xattrs).
  - **Why:** Rollback safety hinges on restoring *exactly* the prior state. A subtly-wrong restore can leave OpenClaw in a worse or non-starting state and can be hard to diagnose under time pressure.
  - **Fix:** Use `rsync` (preferred) or `ditto` with explicit semantics and preserve metadata. Example:
    - Backup: `sudo -u openclaw rsync -a --delete /Users/openclaw/.openclaw/ /Users/openclaw/.openclaw.bak-v2026.2.17/`
    - Restore: `sudo -u openclaw rsync -a --delete /Users/openclaw/.openclaw.bak-v2026.2.17/ /Users/openclaw/.openclaw/`
    - Also: verify ownership afterwards: `sudo chown -R openclaw:staff /Users/openclaw/.openclaw`

- **[F2]**
  - **Severity:** CRITICAL
  - **Finding:** Phase 7 `tee` output filename uses `upgrade-v2024.txt`, which is inconsistent with the 2026 runbook.
  - **Why:** Operators will later look for the wrong artifact, and it can cause confusion during incident review.
  - **Fix:** Rename to `...-upgrade-v2026-2-24.txt` (or similar) and keep a consistent naming convention across steps.

- **[F3]**
  - **Severity:** SIGNIFICANT
  - **Finding:** LaunchAgent domain usage is likely correct (`gui/502`), but the runbook doesn't verify the domain dynamically and assumes UID 502 permanently maps to `openclaw`. If UID changes or the runbook is reused on another host, stop/start will target the wrong domain.
  - **Why:** `launchctl bootout gui/502/...` will fail silently or act on the wrong user session, leaving the gateway running (or not restarted) while the operator believes it is stopped/started.
  - **Fix:** Derive UID at runtime:
    - `OC_UID=$(id -u openclaw)` then use `gui/$OC_UID`.
    - Also add: `sudo launchctl print gui/$OC_UID | grep -q ai.openclaw.gateway` to confirm load state.

- **[F4]**
  - **Severity:** SIGNIFICANT
  - **Finding:** Stop/start logic mixes label vs service-target syntax in a way that may not match how the agent is loaded. `bootout gui/502/ai.openclaw.gateway` assumes the service identifier is `gui/502/ai.openclaw.gateway`, but LaunchAgents are often referenced as `gui/<uid>/<label>` *only when loaded that way*. If it was bootstrapped from a path, the more reliable approach is bootout by plist path.
  - **Why:** A "bootout succeeded" message isn't guaranteed; the process may remain running, and the subsequent pkill fallback is pattern-based and could miss variations.
  - **Fix:** Use path-based bootout for symmetry with bootstrap:
    - Stop: `sudo launchctl bootout gui/$OC_UID /Users/openclaw/Library/LaunchAgents/ai.openclaw.gateway.plist`
    - Start: `sudo launchctl bootstrap gui/$OC_UID /Users/openclaw/Library/LaunchAgents/ai.openclaw.gateway.plist`
    - And add a `launchctl print` check after each.

- **[F5]**
  - **Severity:** SIGNIFICANT
  - **Finding:** Phase 4 manual config edits are not atomic and not validated as JSON. A partial edit or trailing comma will brick startup.
  - **Why:** This is a high-likelihood operational failure mode, especially under time pressure.
  - **Fix:** Recommend scripted edits + validation:
    - Use `jq` to modify/merge keys and write to a temp file, then `mv` into place.
    - Add `jq . /Users/openclaw/.openclaw/openclaw.json >/dev/null` as a gate before restart.

If an operator pastes/replaces, they may inadvertently delete other hardening fields.
  - **Why:** This could weaken the security posture or change behavior.

  - **Fix:** Add a concrete procedure to fetch and validate the chat ID (e.g., via a small "getUpdates"/bot debug command if OpenClaw exposes it, or via an internal helper script), and add a required "send test heartbeat now" step *before* declaring success.

## Answers to specific reviewer questions

1. **Is the rollback procedure safe? Are there scenarios where it could leave the system in a worse state?**
   **Mostly safe in intent, but not fully safe as written.** Main "worse state" scenarios:
   - Backup/restore via `cp -r` can restore incorrectly (nesting/metadata) (see F1).
   - Rolling back the package version does not fully revert the global Node module tree to a known-good SBOM (see F9).
   - If config file was edited and becomes invalid JSON, rollback restores `.openclaw`, but if the backup itself was incomplete/incorrect due to `cp -r` semantics, you can end up worse off.
   **Fix:** Switch to `rsync -a --delete` (or `ditto`) and add a rollback validation checklist (version, status, port binding, Telegram test, isolation tests, plus SBOM diff if feasible).

2. **Are the three HIGH-risk items (heartbeat DM, safeBinTrustedDirs, gateway auth) correctly identified and mitigated?**
   - **Heartbeat DM:** Correctly flagged as high risk *given the claim*, but mitigation needs stronger "how to get group chat id + permission + test" steps (F7) and the claim itself needs grounding (F14).
   - **safeBinTrustedDirs:** Correctly flagged high risk and the mitigation (adding `/opt/homebrew/bin`) is plausible; however, it should be merged carefully with existing exec config (F6) and ideally verified with an explicit known failing command + expected error text (F15).
   - **Gateway auth key migration:** Risk level being "MEDIUM" in the table seems more accurate than "HIGH" (your text says "three HIGH-risk items ... gateway auth" but the table marks it MEDIUM). Mitigation via `doctor --fix` is reasonable, but you should explicitly verify the effective auth config key after doctor (e.g., grep the config) and have a direct loopback auth test if Telegram is down.

3. **Is the Phase 4 manual config change approach sound, or should these be scripted for atomicity?**
   Manual is workable but **scripted is strongly preferable** because these changes are mandatory and error-prone. At minimum: validate JSON (`jq`) before restart. Best: use `jq` to apply each change to a temp file and atomic `mv` into place (F5), plus preserve existing objects (F6).

4. **Are there any missing verification steps that could catch a silent failure?**
   Yes--key additions:
   - **Log inspection step** after restart: `log show` / gateway logs tail for errors about exec trust, heartbeat delivery, pairing required, JSON parse failure.
   - **Heartbeat explicit end-to-end test** that confirms message arrival in the *new* target (not just "trigger a heartbeat check") and records the chat ID used.
   - **Exec allowlist test** should include both `/usr/bin` and Homebrew binaries and confirm the *error string* when blocked (to distinguish PATH vs trust-dir vs workspaceOnly issues).
   - **Config validity check** (`jq .`) before restart (F5).
   - **launchctl state check** (`launchctl print gui/$UID/...`) to ensure the service is actually loaded/unloaded (F4).

5. **Given LaunchDaemon is disabled and LaunchAgent is the active supervisor, is the stop/start procedure using the correct launchctl domain (gui/502)?**
   **Likely yes for this specific host/user mapping**, and the runbook is consistent about `gui/502`. The improvement is to **derive UID dynamically** and prefer **path-based bootout** to avoid label/target mismatches (F3/F4). This makes the runbook robust if UID changes or if the service was loaded in a different way.

**[F2] [SIGNIFICANT] JSON Configuration Integrity Risk (Phase 4)**
- **Finding:** Phase 4 instructs the user to "Add to `openclaw.json`" and provides partial JSON snippets (e.g., just the `tools` or `agents` objects).
- **Why:** Manually merging partial JSON snippets into an existing configuration file is error-prone. A user might overwrite existing keys (e.g., wiping existing `tools` settings while adding `exec`) or create invalid JSON syntax (trailing commas, missing braces), which would cause the service to fail on restart.
- **Fix:** Use `jq` (or a node script) to programmatically merge the new keys, or explicitly instruct the user to use a JSON-aware editor to *merge* keys rather than append/overwrite.
  *Example:* `tmp=$(mktemp) && jq '.tools.exec.safeBinTrustedDirs = ["/bin", "/usr/bin", "/opt/homebrew/bin"]' ~/.openclaw/openclaw.json > "$tmp" && mv "$tmp" ~/.openclaw/openclaw.json`

**[F3] [SIGNIFICANT] Missing Prerequisite: Telegram Group ID**
- **Finding:** Phase 4b requires the user to configure a `target` (group chat ID) for heartbeats. The runbook notes "if no Telegram group exists... create one," but this is a blocking action inside the critical path of the upgrade window.
- **Why:** Attempting to retrieve a Chat ID often requires interacting with the bot API or using a raw update viewer. Doing this while the gateway is stopped (Phase 1) might be impossible if the bot requires the gateway to function to echo its Chat ID.
- **Fix:** Move the acquisition of the Telegram Group ID to the **Pre-Upgrade Checklist**. Verify the ID is known and recorded before stopping the gateway.

**[F4] [MINOR] Rollback Dependency on External Registry**
- **Finding:** The Rollback procedure (Step 3) uses `npm install -g openclaw@2026.2.17`.
- **Why:** If the upgrade failed due to network issues, or if the package registry is unavailable, the rollback will fail, leaving the system with no installed version. The file backup covers data (`.openclaw`), but not the binary/package code.
- **Fix:** In the Pre-Upgrade Checklist, locate the current binary path or pack the current version to a tarball (e.g., `npm pack`) to ensure a local restore source is available without network access.

**[F5] [STRENGTH] Correct Launchctl Domain Usage**
- **Finding:** The runbook correctly identifies the target supervisor as a LaunchAgent and uses `gui/502` for the `bootout` and `bootstrap` commands.
- **Why:** Many runbooks incorrectly mix `system/` (LaunchDaemon) and `gui/` (LaunchAgent) contexts. Using `gui/502` ensures the process runs in the correct user session context, which is vital for the Mac Studio environment described.
