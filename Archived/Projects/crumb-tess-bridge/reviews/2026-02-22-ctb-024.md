# Peer Review: CTB-024 Structured Escalation Flow

**Artifact:** dispatch_engine.py, dispatch_state.py, brief_builder.py, bridge-watcher.py, test_dispatch_engine.py
**Mode:** full
**Reviewed:** 2026-02-22
**Reviewers:** GPT-5.2, Gemini 3 Pro Preview (truncated), DeepSeek Reasoner, Grok 4.1 Fast Reasoning
**Review prompt:** Security-focused review of CTB-024 structured escalation implementation — 5 files (~5,900 LOC), evaluated against dispatch-protocol.md §6 design reference. Questions covered: regex bypass vectors, race conditions in timeout/resume, trust boundary labeling, answer resolution injection resistance, test coverage gaps.

## OpenAI (GPT-5.2)

- [F1]
- [Severity]: SIGNIFICANT
- [Finding]: Escalation regexes are ASCII-allowlist but still allow newline-free "prompt injection by semantics" (e.g., `Ignore previous`, `SYSTEM:`, `###`, etc.) and do not explicitly forbid leading/trailing whitespace, repeated spaces, or ambiguous punctuation patterns; they also don't constrain `escalation_id`/question `id` format at all.
- [Why]: §6.2/§9.3 are primarily about *injection surface reduction*. Character allowlists help, but attackers can still craft high-impact instructions using only allowed characters (letters, spaces, punctuation). Unconstrained IDs can be abused for log/format confusion, key collisions in downstream consumers, or weird rendering issues (e.g., very long IDs or IDs with separators if later added). While Telegram formatting injection is mostly mitigated by the allowlist (no `_ * [ ]` etc.), semantic injection into the resume prompt remains possible because context/questions are embedded verbatim.
- [Fix]:
  1) Add strict ID regex + length caps, e.g. `^[a-z][a-z0-9_-]{0,31}$` for question IDs and `^[a-z0-9]{8,64}$` (or UUIDv4) for `escalation_id`.
  2) Normalize and validate whitespace: `.strip()` and reject if it changes; optionally collapse internal multiple spaces.
  3) Consider tightening text further for "trusted-resolution embedding": forbid substrings like `system:`, `assistant:`, `developer:`, `ignore previous`, `begin`, `end`, `json`, etc. (yes, this is heuristic, but it meaningfully reduces semantic injection).
  4) Alternatively (cleaner): do **not** embed free-form context/questions into the resume prompt; embed only stable identifiers + selected option text, and keep context for Telegram only.

- [F2]
- [Severity]: SIGNIFICANT
- [Finding]: `_build_escalation_resolution` treats `context` as "TRUSTED" in the header, but `context` ultimately originates from an untrusted stage output; it is only regex-validated, not provenance-trusted.
- [Why]: This is a trust-boundary labeling bug. The resume prompt explicitly signals to the model that the block is "TRUSTED, runner-resolved", which can cause the agent to overweight attacker-authored context that passed the regex. Even without newlines, attacker context can still steer decisions.
- [Fix]: Change the resolution block to:
  - Mark only *answer indices → option text mapping* as trusted.
  - Treat `context` and question text as "UNTRUSTED (model-provided), validated for charset only".
  - Or omit context entirely from the resume prompt (preferred), or include it under an `[Untrusted escalation context]` label.

- [F3]
- [Severity]: MINOR
- [Finding]: Regexes use `re.match(...)` with `^...$`, which is fine, but they use `.match` not `.fullmatch`; future edits to the pattern could accidentally remove anchors and reintroduce partial matches.
- [Why]: This is a maintainability footgun in a security boundary.
- [Fix]: Replace with `pattern.fullmatch(text)` for option/text/context validation.

- [F4]
- [Severity]: STRENGTH
- [Finding]: Answer resolution uses index→runner-persisted option text and does not incorporate raw user-provided option strings.
- [Why]: This directly implements §6.5's injection resistance goal: even if the Telegram user (or adversary) sends manipulated option text, only the persisted `options[index-1]` is injected into the resume prompt.

- [F5]
- [Severity]: SIGNIFICANT
- [Finding]: Confirm answers pass through raw `'yes'/'no'` without being resolved from persisted options (though normalized/lowercased). Also, the escalation-response schema isn't validated to be "indices only" for all question types (it allows any value coercible to int for choice; any string for confirm that lowercases to yes/no).
- [Why]: Confirm is low-risk compared to free text, but it's still "raw user input" being injected into the prompt. Today the domain is tiny (`yes|no`), so it's safe; however, the code's trust story is inconsistent and could regress if confirm options ever expand beyond yes/no.
- [Fix]: Treat confirm exactly like choice: require answers to be an index (1/2) and resolve via persisted `["yes","no"]`. Enforce answer object schema strictly (ints only).

- [F6]
- [Severity]: SIGNIFICANT
- [Finding]: Timeout scanning in `BridgeWatcher` claims "no flock needed — blocked dispatches have no competing writers", but there *is* a competing writer: `handle_escalation_response` can transition the same dispatch from blocked→running at the same time the watcher transitions blocked→failed on timeout.
- [Why]: This is a race that can produce inconsistent final states and duplicate/out-of-order user-visible responses (timeout failure vs resumed completion). Atomic file writes don't prevent TOCTOU: both actors read "blocked", then both write different terminal/nonterminal states.
- [Fix]:
  1) Acquire the same global lock/flock used for dispatch operations during `check_escalation_timeouts`, **or** implement per-dispatch lock files.
  2) Add a compare-and-swap style check in `DispatchState.transition`: include an `expected_current_state` and fail if state changed since read.
  3) If keeping watcher lock-free, at least re-load state immediately before transitioning and ensure it is still `blocked` *and* escalation_id still matches.

- [F7]
- [Severity]: SIGNIFICANT
- [Finding]: Resume path reconstructs `current_stage` from `state.get("stage_counter", 0)` and then `_run_stages` immediately does `current_stage += 1`. If `stage_counter` semantics are "completed stages" vs "current stage number", this can off-by-one and re-run or skip a stage. Also, `blocked_stage_output_next` is set to `None` regardless of prior `next_stage` instructions.
- [Why]: Stage numbering errors can cause repeated execution (dangerous for side effects) or missing continuation context. Dropping `next_stage_instructions` can cause the agent to proceed without the intended handoff/instructions after human input.
- [Fix]:
  1) Explicitly persist the blocked stage number in state (e.g., `blocked_stage_number`) and resume with `current_stage = blocked_stage_number` such that the next stage is `blocked_stage_number + 1`.
  2) Persist and reload `next_stage_instructions` at block time (or store the blocked stage's `next_stage.instructions` in `pending_escalation`), then pass it on resume.

- [F8]
- [Severity]: MINOR
- [Finding]: `handle_escalation_response` ignores malformed `timeout_at` parsing errors (`except ...: pass`) and proceeds (accepts response) even if timeout data is corrupted.
- [Why]: If state corruption occurs, accepting late responses may violate §6.6 expectations and produce odd lifecycle behavior.
- [Fix]: On parse failure, treat as timed-out or fail closed: reject response with an error and require a new dispatch/escalation.

- [F9]
- [Severity]: MINOR
- [Finding]: Escalation timeout scanning instantiates a new `DispatchEngine` every watcher cycle, re-importing and rebuilding dependencies.
- [Why]: Not a correctness bug, but it's unnecessary overhead in a tight kqueue loop; could increase latency or CPU wakeups.
- [Fix]: Create and reuse a single engine instance in the watcher, or move timeout scanning into the existing dispatch processor with shared config/engine.

- [F10]
- [Severity]: STRENGTH
- [Finding]: Tests include dedicated categories for escalation validation, answer resolution, flow lifecycle, and timeout scanning.
- [Why]: The right security-critical units are being exercised, reducing regression risk.

- [F11]
- [Severity]: SIGNIFICANT
- [Finding]: Test coverage likely misses the race between timeout scanner and escalation-response resume (the most security/UX impactful failure mode) and the "trust-labeling" issue (context marked trusted).
- [Why]: The race is subtle and won't appear in deterministic unit tests unless explicitly modeled; yet it can cause contradictory terminal states/responses in production.
- [Fix]: Add concurrency/TOCTOU tests (even if simulated) that:
  1) Set state to blocked with imminent timeout; concurrently call `check_escalation_timeouts()` and `handle_escalation_response()`; assert only one wins and the other is rejected based on current state/escalation_id.
  2) Assert `_build_escalation_resolution` labeling does not mark untrusted fields as trusted (or assert context omitted).

- [F12]
- [Severity]: MINOR
- [Finding]: Escalation schema validation strips `default` by omission (good), but it doesn't explicitly reject unknown extra keys in escalation/question objects.
- [Why]: Extra keys can become a future injection vector if later code starts using them implicitly (e.g., `label`, `markdown`, `telegram_parse_mode`).
- [Fix]: Enforce an allowlist of keys for escalation and questions; reject if unknown keys exist.

### 1. Escalation Validation and Regex
**[F1] [STRENGTH] Strict Whitelist Regex Validation**
The implementation uses strict regex allowlists (`^[A-Za-z0-9 ,.;:!?'()-]+$`) for all textual inputs (context, question text, options). ## DeepSeek (DeepSeek Reasoner)

- [F1]
- [Severity]: MINOR
- [Finding]: Escalation validation regex patterns do not escape hyphens in character class ranges, potentially causing interpretation issues
- [Why]: The regex `r"^[A-Za-z0-9 ,.;:!?'()-]{1,80}$"` places hyphen at the end of character class where it could be interpreted as part of a range (like `'-'`). While Python's regex engine handles this correctly when hyphen is at start or end of class, it's a best practice to escape it
- [Fix]: Change regex patterns to escape hyphens: `r"^[A-Za-z0-9 ,.;:!?'()\-]{1,80}$"` and similarly for other patterns

- [F2]

- [Finding]: Escalation timeout comparison may fail for timezone-naive datetime strings
- [Why]: In `handle_escalation_response`, timeout comparison converts `timeout_at_str` to datetime and adds UTC if naive, but `blocked_at` parsing lacks same robustness. If `blocked_at` is naive and system timezone isn't UTC, `blocked_seconds` calculation could be wrong
- [Fix]: Apply consistent timezone handling: wrap both conversions in a helper function that ensures UTC timezone

- [F3]
- [Severity]: SIGNIFICANT
- [Finding]: Missing validation for `escalation_id` format in `_validate_escalation`
- [Why]: `escalation_id` is validated as non-empty string but not checked for length or safe characters. This ID appears in relay filenames and state

## Synthesis

### Consensus Findings

**1. Race condition: timeout scanner vs. escalation response resume (CRITICAL)**
Flagged by: OAI-F6, DSK-F4, GRK-F2

All three reviewers independently identified the same TOCTOU race: `handle_escalation_response` and `check_escalation_timeouts` can both read "blocked" state simultaneously, then both attempt transitions (one to "running", one to "failed"). The flock is released while a dispatch is blocked (by design — Option A), leaving no mutual exclusion between these two writers. Atomic JSON writes don't prevent the race; last writer wins.

**2. Missing regex validation on `escalation_id` and question `id` fields (CRITICAL)**
Flagged by: OAI-F1 (partial), DSK-F3, GRK-F1

These IDs are validated for type (string) and non-emptiness, but not character set or length. Since they're embedded in relay filenames, state JSON, and the resume prompt, control characters (newlines, tabs) or excessively long IDs could enable prompt injection or file system issues. The text fields (options, context, question text) have regex validation, but IDs were overlooked.

**3. Trust-boundary labeling: "TRUSTED" label on untrusted content (SIGNIFICANT)**
Flagged by: OAI-F2, GRK-F3

`_build_escalation_resolution` marks the entire escalation resolution block as "TRUSTED, runner-resolved", but `context` and question `text` originate from untrusted stage output. Only the answer→option-text mapping is genuinely runner-resolved. The "TRUSTED" label may cause the LLM to overweight adversarial context that merely passed regex validation.

**4. Watcher instantiates DispatchEngine every cycle (MINOR)**
Flagged by: OAI-F9, DSK-F8, GRK-F5

`_check_escalation_timeouts` creates a new `DispatchEngine` instance (including `sys.path.insert`) on every watcher cycle. Functionally correct but wasteful.

**5. `next_stage_instructions` dropped on resume (SIGNIFICANT)**
Flagged by: OAI-F7, DSK-F11

When resuming from blocked state, `blocked_stage_output_next` is set to `None`, discarding any `next_stage.instructions` the blocked stage may have produced. This could cause the agent to proceed without intended handoff context.

**6. Answer resolution for index-to-text is injection-resistant (STRENGTH)**
Flagged by: OAI-F4, DSK-F7, GRK-S1, GEM-F3

All four reviewers confirmed that `_resolve_escalation_answers` correctly maps indices to runner-persisted option text per §6.5. Raw user input is never injected into the resume prompt for choice questions.

**7. Comprehensive test coverage (STRENGTH)**
Flagged by: OAI-F10, DSK-F12, GRK-S1 (partial)

46 new tests across 5 well-structured categories. Right security-critical units are exercised.

### Unique Findings

**OAI-F3 — `re.match` vs `re.fullmatch` (MINOR):** Genuine maintainability concern. The current patterns have `^...$` anchors so behavior is correct today, but `fullmatch` is more idiomatic and defensive. *Likely genuine insight.*

**OAI-F5 — Confirm answers as raw strings vs indices (SIGNIFICANT):** Interesting consistency argument. `yes`/`no` is a closed domain so injection risk is near-zero, but the trust model is inconsistent with choice questions. *Genuine insight, low urgency.*

**OAI-F8 — Silent `except: pass` on timeout_at parse failure (MINOR):** Accepting responses when timeout data is corrupted could produce odd lifecycle states. *Genuine insight.*

**OAI-F12 — No rejection of unknown extra keys in escalation schema (MINOR):** Defense in depth concern. Extra keys are currently ignored but could become injection vectors if future code starts consuming them. *Genuine insight.*

**DSK-F2 — Timezone-naive datetime inconsistency (SIGNIFICANT):** `timeout_at` parsing adds UTC if naive, but `blocked_at` parsing may not apply the same treatment. If system timezone isn't UTC, `blocked_seconds` calculation drifts. *Genuine insight — needs verification against actual code.*

**DSK-F5 — String '1' vs integer 1 for choice answers (SIGNIFICANT):** Claims `int(answer)` rejects string '1'. *Needs verification* — `int("1")` succeeds in Python, so this may be a false finding. If the code does `isinstance(answer, int)` check first, this is valid.

**DSK-F9 — No rejection of extra answer keys (SIGNIFICANT):** Extra answers beyond the question set are silently ignored. *Genuine insight, low urgency.*

**DSK-F10 — Predictable relay file naming (SIGNIFICANT):** Relay files use `{dispatch_id}-escalation.json`. *Noise* — this is a local single-user system where dispatch_id is already known to the dispatch creator. File enumeration is not a threat in this architecture.

**GRK-F4 — Path traversal in deliverables reconstruction (SIGNIFICANT):** Malicious stage output could persist bogus paths in deliverables. *Genuine insight* — though the paths are only used for error response metadata, not file reads, path normalization is still good practice.

**GRK-F7 — Non-unique escalation_id across dispatches (MINOR):** Per-dispatch scoping makes cross-dispatch collision harmless. *Noise for current architecture.*

**GEM-F2 — No-newline regex blocks multi-line questions (MINOR):** Correct observation but this is by design per §6.2. Agent must produce single-line questions. *Expected behavior, not a bug.*

### Contradictions

**Regex strictness — tighten vs. sufficient:**
- OAI-F1 and GRK-F3 argue the regex should be tighter (drop `!?'`, add semantic keyword blocklists)
- GEM-F1 considers the current regex a STRENGTH that "effectively neutralizes injection attacks"
- DSK-F1 focuses on hyphen escaping (cosmetic)

*Assessment:* The regex is a reasonable balance for §6.2 compliance. Semantic injection via allowed characters is theoretically possible but extremely constrained (no newlines, no brackets, no quotes beyond single-quote, 80-200 char limits). The real fix is the trust labeling (finding #3 above), not further regex tightening.

**Confirm answer handling — raw vs indexed:**
- OAI-F5 argues confirm should use indices like choice (consistency)
- DSK-F7 and GRK-S1 consider current yes/no handling correct and injection-resistant

*Assessment:* Both positions are valid. Current implementation is safe (two-element closed domain), but the inconsistency is a code smell. Deferring — low risk.
