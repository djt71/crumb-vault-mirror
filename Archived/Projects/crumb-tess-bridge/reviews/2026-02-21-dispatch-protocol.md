# Peer Review: Dispatch Protocol (CTB-016)

**Artifact:** `Projects/crumb-tess-bridge/design/dispatch-protocol.md`
**Mode:** full
**Reviewed:** 2026-02-21
**Reviewers:** GPT-5.2, Gemini 3 Pro Preview, DeepSeek Reasoner (V3.2-Thinking), Grok 4.1 Fast Reasoning
**Review prompt:** 7 specific review questions (correctness, security, completeness, feasibility, schema coherence, operability, escalation safety) plus context on existing bridge architecture and threat model.

---

## OpenAI (GPT-5.2)

- [F1]
- [Severity]: CRITICAL
- [Finding]: `running → blocked` transition is impossible with the current file model (stage output is only read after the stage exits).
- [Why]: The protocol says "running → blocked: Stage output declares blocked," but stage output is written to outbox and then read by the runner after the process exits. That means "blocked" can only be observed at `stage-complete`, not while `running`. This creates ambiguity for status reporting, escalation timing, and timeout semantics.
- [Fix]: Redefine the transition as `running → stage-complete → blocked`. Update §2.3 rule 6 accordingly and specify that "blocked" is entered only after runner parses the completed stage output. If you want true mid-stage blocking, add an explicit *checkpoint output* channel (e.g., `_openclaw/outbox/{dispatch}-checkpoint.json`) that the runner can poll while the session is alive.

---

- [F2]
- [Severity]: CRITICAL
- [Finding]: "Discard ALL stage deliverables on governance mismatch" is not actually enforceable in a file-writing model where the stage may have already modified the vault.
- [Why]: Claude Code can write/modify files during the stage. If post-stage governance verification fails, the runner can refuse to continue, but it cannot reliably undo writes. The doc promises rollback-like behavior ("discard deliverables"), which will be false in practice and undermines the security story.
- [Fix]: Change the guarantee: on governance mismatch, (a) halt dispatch, (b) alert, (c) quarantine by recording a manifest of modified files and optionally revert via VCS if available. Stronger option: require all writes to occur via a runner-controlled "apply" step (patches generated by the stage, applied by runner) or run stages in a disposable workspace and only copy results into the vault after verification.

---

- [F3]
- [Severity]: CRITICAL
- [Finding]: Budget enforcement trusts self-reported `metrics` from the untrusted worker.
- [Why]: Tool calls/tokens/wall-time in `stage output.metrics` can be omitted, falsified, or manipulated by a compromised/confused session, bypassing budget checks and "security boundary" claims (§8.4). Even honest sessions may not have accurate token counts.
- [Fix]: Make budgets runner-observed wherever possible:
  - Wall time: measured by runner (start/stop timestamps), not from JSON.
  - Stages: runner-counted.
  - Tool calls: ideally from Claude Code logs/telemetry if available; otherwise treat worker-reported tool_calls as *advisory* and enforce a conservative stage cap + wall-time cap as the true boundary.
  - Tokens: if you can't get authoritative counts, remove it as an enforcement dimension (keep as reporting-only), or estimate from transcript size with conservative multipliers and hard-stop earlier.

---

- [F4]
- [Severity]: SIGNIFICANT
- [Finding]: Governance verification fields are provided by the worker, but there's no described binding between "this stage output" and "the actual CLAUDE.md that governed the session."
- [Why]: You compute hash/canary in runner and compare to stage output. But a malicious/buggy worker can echo expected values without actually being governed (especially if something about invocation or tool restriction fails). The scheme reduces to "worker claims it loaded governance."
- [Fix]: Add a runner-verifiable proof tied to the invocation, e.g.:
  - Runner injects a nonce (already mentioned) and requires the stage output to include `nonce` and an HMAC/signature computed by a runner-held secret (best) or by reading a runner-provided one-time token file (second-best).
  - Or: runner appends a canary string *only present in the system prompt* and requires it to be reproduced verbatim in output (still spoofable, but at least ties to prompt receipt).
  - Also specify *how* `claude --print` is invoked to ensure `--append-system-prompt` actually loads CLAUDE.md and can't be bypassed by stage instructions.

---

- [F5]
- [Severity]: SIGNIFICANT
- [Finding]: Dynamic stage sequencing allows a stage to smuggle unbounded instructions into `next_stage.instructions`, defeating the "runner is mechanical" safety posture.
- [Why]: Even with budgets, a malicious stage can issue instructions that cause the next stage to exfiltrate data, touch sensitive files, or perform risky operations—without any additional runner scrutiny. "Runner MUST NOT interpret or modify stage instructions" turns the worker into the de facto orchestrator.
- [Fix]: Introduce a small, generic runner-side policy layer that is still "mechanical":
  - Enforce maximum length/charset constraints on `next_stage.instructions`.
  - Enforce an allowlist of referenced `context_files` locations (e.g., within the vault; exclude `~/.ssh`, `~/.aws`, etc.).
  - Require next stage to declare an `intent` + `risk_tier` (enumeration) and have runner block/force escalation when it increases.
  - Consider restricting worker to output *structured next actions* (e.g., `action_type: read_files|write_files|summarize|draft_patch`) rather than freeform instructions.

---

- [F6]
- [Severity]: SIGNIFICANT
- [Finding]: Escalation "no free-text" reduces injection risk, but options can still be engineered to induce harmful choices or encode instructions indirectly.
- [Why]: A compromised session can present choices like "Option A: Proceed (recommended)" / "Option B: Cancel" with misleading context, social engineering the user. Also "no file paths in options" is stated, but not enforceable unless runner validates it; and even without paths, options can encode operational instructions ("Use the admin key").
- [Fix]: Add runner-side validation + UX hardening:
  - Runner rejects options containing forbidden patterns (slashes, `..`, `~`, backticks, "rm", "sudo", etc.) and enforces a strict regex for options (e.g., `^[A-Za-z0-9 ,.'-]{1,80}$`).
  - Tess displays *neutral* labels (A/B/C) and hides worker-supplied "recommended/default" unless it's from a trusted policy module.
  - For `risk` gate, require confirm question text to include a fixed runner-generated warning and require a second confirmation (two-step).

---

- [F7]
- [Severity]: SIGNIFICANT
- [Finding]: Escalation response answers include "selected option text," which can reintroduce prompt injection if that text is inserted into the next stage prompt.
- [Why]: Even if options are short, inserting raw option strings into the resume prompt is still an untrusted string from the worker (because the worker authored the options). That's effectively a worker→worker communication channel that bypasses other constraints.
- [Fix]: In escalation-response, store/forward the *option index* (1/2/3/4) or an option `value` that is runner-generated. Runner should persist the original option list from the blocked stage output and resume with "User selected option #2" plus the runner-stored canonical option text (after validation/sanitization).

---

- [F8]
- [Severity]: SIGNIFICANT
- [Finding]: Lifecycle does not define crash recovery or idempotency; "ledger not persisted; crash => fail" is operationally harsh and can strand `blocked` dispatches.
- [Why]: Runner restarts (OS update, crash) will "fail" a dispatch even if it's safely resumable, and you may lose the association between a pending escalation and later user response. Also outbox status overwrites can hide what happened.
- [Fix]: Persist minimal dispatch state to disk (`_openclaw/dispatch/{dispatch_id}.json`):
  - current state, stage counter, budgets consumed (runner-measured), pending escalation_id, and paths to produced artifacts
  - on startup, runner can mark `running` as `failed` with `RUNNER_CRASHED` but allow `blocked` to remain resumable, or allow explicit "resume" op.

---

- [F9]
- [Severity]: SIGNIFICANT
- [Finding]: Concurrency/rate limiting is underspecified and slightly inconsistent ("flock + max 2 concurrent" vs DISPATCH_CONFLICT).
- [Why]: If flock enforces a single global lock, you can't have 2 concurrent dispatches. If you allow 2 concurrent, you need per-dispatch locks plus a scheduler. Ambiguity impacts correctness and DoS handling.
- [Fix]: Specify the concurrency model precisely:
  - Option A (simplest): single-dispatch global lock; remove "max 2 concurrent" language.
  - Option B: allow N concurrent; implement a queue + per-dispatch state files + worker pool; define fairness and per-project limits.

---

- [F10]
- [Severity]: MINOR
- [Finding]: State machine lacks an explicit `canceled` terminal state; cancellation is implied ("CANCEL {dispatch_id_short}") but not defined in lifecycle or error codes.
- [Why]: Operators need deterministic handling for user cancel, plus audit clarity and correct cleanup.
- [Fix]: Add `canceled` as a terminal state and an operation `cancel-dispatch` (or treat as `failed` with code `CANCELED_BY_USER`). Define behavior mid-stage (cannot interrupt) vs between stages (stop before next spawn).

---

- [F11]
- [Severity]: SIGNIFICANT
- [Finding]: "Stage outputs — summaries only" as inter-stage context is likely insufficient for feasibility; many tasks need precise state (filenames, decisions, partial results).
- [Why]: With isolated sessions, if you only pass summaries, later stages may repeat work, miss constraints, or regress decisions. Passing full transcripts is unsafe/noisy; passing too little harms success rate.
- [Fix]: Add a structured, bounded "stage memory" object:
  - `handoff`: small JSON with decisions, key facts, and pointers to files created
  - enforce size limits (e.g., 8–16KB) and require runner to include it verbatim
  - require any large context to be written to a vault file and referenced via `context_files`.

---

- [F12]
- [Severity]: SIGNIFICANT
- [Finding]: Status update file is overwritten; if Tess misses a filesystem event, progress can be lost.
- [Why]: Telegram relay reliability depends on event watching. Overwrite-only reduces traceability and makes debugging hard ("what happened between stage 2 and 4?").
- [Fix]: Keep overwrite for "latest," but also append an immutable log: `_openclaw/outbox/{dispatch_id}-status-{timestamp}.json` or `_openclaw/dispatch/{dispatch_id}/status.log.jsonl`. Tess can still only message once per stage, but you retain full history.

---

- [F13]
- [Severity]: MINOR
- [Finding]: Schema coherence: `schema_version: 1.1` is reused while making additive changes, but you're also introducing entirely new message types and operations without a clear compatibility story.
- [Why]: If older components expect 1.1 but don't know `escalation-response` or new outbox files, behavior may degrade silently.
- [Fix]: Either bump to `1.2` for the dispatch extension or define "1.1 + dispatch extension" capability negotiation (e.g., runner writes `_openclaw/capabilities.json` and Tess checks).

---

- [F14]
- [Severity]: SIGNIFICANT
- [Finding]: Audit hash chaining is underspecified and may be unstable across platforms (newline normalization, metadata, ordering).
- [Why]: If the hash is used for tamper detection, it must be reproducible deterministically.
- [Fix]: Specify:
  - exact hash input = sha256 of raw bytes of each transcript file
  - ordering = by stage_number ascending
  - concat format = hex hashes joined with `\n` (LF) and final LF
  - include stage output JSON hashes too, if they matter.

---

- [F15]
- [Severity]: STRENGTH
- [Finding]: Clear separation of roles (runner as orchestrator, Tess as transport, Crumb as worker) and stage isolation with per-stage governance checks is a strong architectural direction.
- [Why]: This reduces long-session drift, creates natural checkpoints, and gives the operator visibility and control points (budgets, kill-switch, escalation).
- [Fix]: None.

---

- [F16]
- [Severity]: STRENGTH
- [Finding]: Structured escalation + confirmation echo is a solid defense-in-depth pattern against NLU misparse and accidental resume with wrong answers.
- [Why]: It materially reduces "silent wrong action" risk and makes the human explicitly accountable at decision points.
- [Fix]: Extend with the option-index fix in F7 and stronger option validation in F6.

---

- [F17]
- [Severity]: SIGNIFICANT
- [Finding]: Operability/debugging story needs explicit "single place to look" and correlation across artifacts.
- [Why]: With many files (stage outputs, transcripts, status, final response), solo-operator debugging becomes tedious without an index/manifest.
- [Fix]: Have runner write `_openclaw/dispatch/{dispatch_id}/manifest.json` containing:
  - brief, budgets, per-stage timestamps, exit codes, transcript paths, deliverable manifest, governance results, and any escalation Q/A.
  - Include a "last_error" field and pointers to relevant logs.

---

- [F18]
- [Severity]: SIGNIFICANT
- [Finding]: "Runner MUST NOT interpret stage instructions" conflicts with security needs around filesystem boundaries and sensitive paths.
- [Why]: Without any interpretation, the runner can't enforce "context_files must be safe," can't prevent reading secrets, and can't enforce risk tiers mechanically.
- [Fix]: Reframe: runner must not do *semantic planning*, but it must enforce *policy validation*. Define a small set of mechanical checks (path allow/deny lists, size limits, regex constraints, risk-tier monotonicity) that are explicitly permitted.

---

- [F19]
- [Severity]: MINOR
- [Finding]: Escalation timeout default (30 minutes) + "fail and resubmit to resume" may cause repeated work and user frustration.
- [Why]: Many escalations happen when user is away. Immediate failure also loses the ability to cleanly resume with preserved state.
- [Fix]: Consider transitioning to `blocked` with `expired: true` but not terminal; allow `escalation-response` later to resume, or implement an explicit "resume-dispatch" that reopens within a longer TTL (e.g., 24h) while still preventing indefinite resource holding.

---

- [F20]
- [Severity]: SIGNIFICANT
- [Finding]: Feasibility: writing transcript to a path assumes Claude Code can be reliably instructed to write files deterministically; failure modes (partial JSON, extra text) aren't handled.
- [Why]: `claude --print` may emit additional stdout text, or the model may fail to write valid JSON. Without robust parsing and retries, dispatches will frequently fail.
- [Fix]: Add runner handling rules:
  - Stage output must be read from a specific file; runner ignores stdout except for logging.
  - If JSON invalid/missing, runner retries the same stage once with an "output repair" prompt (within budget) or marks `STAGE_FAILED`.
  - Use a strict JSON schema validator and require `content-type` markers if needed.

## Google Gemini 3 Pro Preview

### Executive Summary

The protocol represents a robust, well-structured approach to asynchronous agent orchestration. The "Runner as Orchestrator / Crumb as Worker" model effectively isolates failures and enforcing governance at granular intervals (stages). The decision to avoid a "god-mode" loop in favor of discrete, budget-bounded stages significantly reduces the blast radius of a compromised session.

However, a critical vulnerability exists in the propagation of context between stages (the "Prompt Injection Worm" vector), and the lifecycle is missing a graceful cancellation mechanism for non-blocked tasks.

---

### Findings

#### [F1] CRITICAL | SECURITY | Inter-Stage Prompt Injection ("Worm" Vector)
**Finding:** The protocol feeds the `next_stage.instructions` output from Stage N directly into the prompt of Stage N+1 (Section 4.2). If Stage N ingests malicious data (e.g., a poisoned PDF) that performs a prompt injection, it can force Stage N to output malicious instructions for Stage N+1. Because Stage N+1 receives these as "instructions" from the trusted Runner, it may bypass system prompt protections.
**Why:** This breaks the isolation between stages. A successful injection in a low-privilege stage (e.g., "read-only research") could pivot to a high-privilege stage (e.g., "execute code") by manipulating the `next_stage` instructions.
**Fix:**
1.  **Framing:** The Runner must strictly frame the previous stage's output in the Stage N+1 prompt. Do not label it "Instructions." Label it "Context from previous stage."
2.  **Static System Instructions:** The core instructions for *how to behave* (e.g., "Do not delete files without approval") must be statically injected by the Runner in the system prompt for *every* stage, ensuring they override any dynamic instructions coming from the previous stage.
3.  **Sanitization:** The Runner should enforce a strict allowlist of characters for `next_stage.instructions` or strip formatting that mimics system headers.

#### [F2] SIGNIFICANT | COMPLETENESS | Missing Active Dispatch Cancellation
**Finding:** The State Machine (§2.1) and Escalation (§6.3) allow cancellation only when a dispatch is in the `blocked` state. There is no defined mechanism to cancel a dispatch that is in `queued`, `running`, or `stage-complete` states without using the global Kill-Switch.
**Why:** If a user accidentally dispatches a task with a high budget (e.g., 25 stages) that is going in the wrong direction (visible via status updates), they must wait for it to finish, wait for it to block, or kill the entire bridge.
**Fix:** Add a `cancel-dispatch` operation to the Phase 2 allowlist. The Runner should check for a `cancellation-{dispatch_id}` file in the inbox between stages (at the same time it checks the kill-switch).

#### [F3] SIGNIFICANT | OPERABILITY | Crash Recovery & Zombie Dispatches
**Finding:** Section 8.2 states the budget ledger is "in memory (not persisted to disk)." Section 3.1 implies the runner manages the lifecycle. If the Runner process crashes or the host reboots, the in-memory state is lost.
**Why:** Upon restart, the Runner will not know about the pending/running dispatch. The `flock` will be released by the OS, but:
1.  The user receives no notification of failure.
2.  Tess may still be waiting for a response (orphan state).
3.  Intermediate files remain in the outbox.
**Fix:**
1.  Persist the budget ledger to disk (`_openclaw/state/{dispatch_id}-state.json`) atomically after each stage.
2.  Implement a "Startup Scan" in the Runner: On boot, scan `_openclaw/state/`. If any dispatch is not marked `complete/failed` but has no active process, transition it to `failed` (Error: `RUNNER_RESTART`) and write the final response to notify the user.

#### [F4] MINOR | FEASIBILITY | Context Window Exhaustion
**Finding:** Section 4.2 constructs the prompt using "Previous stage outputs — summaries". For a max-budget dispatch (25 stages), accumulating 24 summaries plus the original brief and current instructions may exceed the context window or degrade model performance (recall issues).
**Why:** Claude Code has a large context, but "needle in a haystack" issues persist. Unbounded context growth increases cost and hallucination risk.
**Fix:** Implement a "Rolling Window" or "Summary Compaction" strategy. E.g., include full summaries for the last 3 stages, and a super-compressed "Master Summary" for stages 1-(N-4).

#### [F5] MINOR | SECURITY | Escalation Option Social Engineering
**Finding:** While §6.2 restricts Escalation to `choice`/`confirm`, the *text* of the options is generated by Crumb. A compromised Crumb could present options like:
*   Q: "Security verification required."
*   Option 1: "Authorize access (Recommended)" -> (Actually maps to `rm -rf /`)
**Why:** The user sees the text, but not the underlying tool call mapping that the *next* stage will perform based on that text. The Runner blindly passes the selected string back to the next stage.
**Fix:**
1.  **Gate Coloring:** Tess should visually color-code the message based on the `gate_type` (e.g., RED for `risk`, YELLOW for `conflict`).
2.  **Prompt Warning:** The "Resume" prompt (§6.5) must explicitly warn the model: "The user selected this option based on the text YOU provided. Ensure your subsequent actions strictly align with that text."

#### [F6] STRENGTH | SECURITY | Audit Hash Chain
**Finding:** The mechanism in §7.3 (`audit_hash`) which chains all stage transcript hashes is a strong non-repudiation feature.
**Why:** It effectively creates a lightweight blockchain-like structure for the session history. If an attacker tries to delete a transcript from Stage 3 where they did something malicious, the hash in the Final Response won't match.

#### [F7] STRENGTH | ARCHITECTURE | Budget as Security Boundary
**Finding:** Using "budget" (§8.4) not just for cost control but as a security boundary (limiting "blast radius") is a highly effective design pattern for autonomous agents.
**Why:** It mitigates the worst-case scenario of an infinite loop or a "runaway" agent modifying thousands of files. The hard caps (e.g., 25 stages) provide a definitive fail-safe that overrides user error.

## DeepSeek Reasoner (V3.2-Thinking)

### [F1]
- **[Severity]**: SIGNIFICANT
- **[Finding]**: State machine has missing transitions for session crashes and mid-stage failures
- **[Why]**: The state machine only shows transitions when Crumb produces valid stage output. If `claude --print` crashes, segfaults, or is killed by OS, there's no defined handling. Similarly, if a stage exceeds budget during execution (not just between stages), there's no mechanism to interrupt it.
- **[Fix]**: Add `running → failed` transition for session exit codes ≠0 or early budget violation detection. Add watchdog timer that can kill long-running stages. Document crash recovery: runner should detect orphaned PID, clean up, move to `failed`.

---

### [F2]
- **[Severity]**: SIGNIFICANT
- **[Finding]**: Concurrent dispatch handling is underspecified for race conditions
- **[Why]**: While flock prevents concurrent execution, the protocol doesn't address:
  1. Multiple dispatches queued simultaneously (priority? FIFO?)
  2. Phase 1 operations interleaved with dispatches (could modify files dispatches depend on)
  3. Two dispatches referencing same project/task ID
- **[Fix]**: Specify queue management (FIFO with timestamp). Add conflict detection: runner should check if any active dispatch is touching overlapping file sets and either block or fail the second. Document that Phase 1 operations during dispatch are "at your own risk."

---

### [F3]
- **[Severity]**: MINOR
- **[Finding]**: Stage-level governance verification has a TOCTOU vulnerability
- **[Why]**: Runner computes governance hash, then spawns session. Between compute and spawn, CLAUDE.md could be modified. Session would run with new governance but runner would compare against old hash. The verification happens post-stage, too late.
- **[Fix]**: Runner should compute hash, immediately spawn session, then immediately recompute hash and compare with first. If mismatch, kill session immediately (SIGTERM). Or use file locking on CLAUDE.md during session.

---

### [F4]
- **[Severity]**: SIGNIFICANT
- **[Finding]**: Escalation answers can bypass structured constraints via option text manipulation
- **[Why]**: While options are ≤80 chars ASCII, a compromised session could craft options like:
  1. "yes-ignore-safety" vs "no" (binary choice with embedded instruction)
  2. "delete_file_x" vs "cancel" (disguised command)
  3. Options that form a sentence when concatenated
  The user sees formatted choices but may not recognize embedded commands.
- **[Fix]**: Add content validation rules: options must be simple noun phrases/verbs, no hyphens/underscores that could form commands, runner validates against allowlist of safe patterns. Add visual warning in Telegram for any option containing slash, dash, or dot.

---

### [F5]
- **[Severity]**: CRITICAL
- **[Finding]**: Budget enforcement can be bypassed via tool call recursion
- **[Why]**: Budget counts tool calls per stage via metrics reported by Crumb. A malicious/compromised session could:
  1. Underreport tool calls in metrics
  2. Use a tool that spawns subprocesses making unlimited calls
  3. Chain tool calls (tool A calls tool B) where only parent is counted
- **[Fix]**: Runner must independently count tool calls via process monitoring (e.g., trace syscalls or count Claude API calls). Budget enforcement must be external, not relying on self-reported metrics. Add hard kill when tool call count exceeds budget regardless of reported metrics.

---

### [F6]
- **[Severity]**: MINOR
- **[Finding]**: Stage context passing is fragile for multi-file deliverables
- **[Why]**: Stage outputs include `deliverables` array but only summaries are passed to next stage. If stage 1 creates 5 files, stage 2 needs to know about all 5, but only gets a summary. Could miss critical context.
- **[Fix]**: Enhance `next_stage.context_files` to be mandatory for any deliverables produced. Runner should automatically include all deliverable paths from previous stage in context_files unless explicitly excluded. Add validation that every created/modified file is either in context_files or marked as "ephemeral."

---

### [F7]
- **[Severity]**: SIGNIFICANT
- **[Finding]**: No mechanism for mid-dispatch cancellation by user
- **[Why]**: Once dispatch starts, user can only:
  - Wait for completion
  - Let it hit budget limit
  - Activate kill-switch (stops all dispatches)
  - Timeout escalation (30min wait)
  No way to say "stop this specific task now."
- **[Fix]**: Add `cancel-dispatch` operation. Runner watches for cancellation file matching dispatch_id. When detected, sends SIGTERM to current stage, transitions to `failed` with `USER_CANCELLED`. Tess command: "CANCEL <dispatch_short>" (already in UI but no backend).

---

### [F8]
- **[Severity]**: MINOR
- **[Finding]**: Schema versioning mismatch between files
- **[Why]**: Stage output, status update, final response all have `"schema_version": "1.1"`. Base bridge schema is also 1.1. If base schema updates to 1.2, dispatch schemas become ambiguous. No forward/backward compatibility story.
- **[Fix]**: Use separate version namespace: `"dispatch_version": "1.0"`. Document that dispatch 1.0 works with bridge schema ≥1.1. Add compatibility matrix in spec.

---

### [F9]
- **[Severity]**: SIGNIFICANT
- **[Finding]**: Debugging story is incomplete for partial failures
- **[Why]**: When dispatch fails at stage 5 of 10, operator has:
  - Stage transcripts 1-4
  - Stage output JSON 1-4
  - No transcript for failed stage 5 (if crash)
  - No clear "what was it trying to do?" context
  - No automatic cleanup of partial files
- **[Fix]**: Add failure post-mortem: runner writes `{dispatch_id}-postmortem.md` with:
  1. Last known stage prompt
  2. Budget state at failure
  3. System context (governance hash, kill-switch status)
  4. Recommendations (increase budget? fix file permissions?)

---

### [F10]
- **[Severity]**: STRENGTH
- **[Finding]**: Audit hash chain is excellent for tamper detection
- **[Why]**: SHA256 chain of all stage transcripts provides cryptographic proof that transcripts haven't been altered post-execution. Combined with governance canary, creates strong integrity guarantee.

---

### [F11]
- **[Severity]**: MINOR
- **[Finding]**: Escalation timeout of 30 minutes is too long for some operations
- **[Why]**: Quick-fix (3 stage, 3 min expected) could be blocked for 30min waiting for user. User might forget, causing resource lock.
- **[Fix]**: Make timeout configurable per operation type: quick-fix=5min, invoke-skill=15min, start-task=30min. Add warning at 80% timeout.

---

### [F12]
- **[Severity]**: SIGNIFICANT
- **[Finding]**: Resource isolation between stages is incomplete
- **[Why]**: Each stage runs in fresh `claude --print` session but shares same:
  - File system (vault)
  - Environment variables
  - Network access
  A compromised stage could plant backdoor files for next stage to find.
- **[Fix]**: Implement stage sandboxing: each stage gets temp workspace, runner copies only allowed context_files into it. Stage writes to temp, runner validates outputs before committing to vault. This matches principle of least privilege.

---

### [F13]
- **[Severity]**: MINOR
- **[Finding]**: Telegram formatting for long stage summaries will be truncated
- **[Why]**: Stage summary ≤500 chars, but Telegram message with operation, stage info, budget could exceed 2000 chars (Telegram limit). No truncation strategy defined.
- **[Fix]**: Add explicit truncation in Tess: summary max 300 chars for Telegram. Use ellipsis. Include "full summary in transcript" note.

## Grok 4.1 Fast Reasoning

- **F1**
  - **Severity**: CRITICAL
  - **Finding**: State machine transitions are incomplete and inconsistent; transition 6 claims `running → blocked` "from stage output", but stage output is only readable *after* `running → stage-complete`. No transition from `stage-complete → blocked`. Additionally, `blocked → running` assumes a single pending escalation, but lacks handling for multiple escalations per dispatch.
  - **Why**: Creates unreachable states (e.g., no path to `blocked` without violating logic) and potential deadlocks if a stage outputs `blocked` status—runner would treat it as `stage-complete` then fail to transition properly. Multi-escalation could lead to race conditions or ignored responses.
  - **Fix**: Revise transitions: Add `stage-complete → blocked` when `status: "blocked"` in stage output. Define at most one pending escalation per dispatch (reject additional `blocked` outputs). Add `blocked → failed` on invalid escalation response.

- **F2**
  - **Severity**: CRITICAL
  - **Finding**: No persistence of dispatch state (e.g., current stage, budget ledger, blocked status); runner holds everything in memory. Runner crash mid-dispatch loses all tracking—next runner invocation can't distinguish queued/running from failed/complete, and files like `-status.json` could trigger duplicate execution.
  - **Why**: Leads to zombie dispatches, double-spending budget, or lost partial work on restart. Breaks completeness for crashes/partial failures, undermining auditability and operability for solo operator.
  - **Fix**: Introduce a persistent state file per dispatch (`{dispatch_id}-state.json`) with atomic writes: fields for `lifecycle_state`, `budget_consumed`, `pending_escalation_id`, `next_stage_n`. Runner reads/writes it on every transition. Lock it with flock.

- **F3**
  - **Severity**: SIGNIFICANT
  - **Finding**: Concurrent dispatches not fully handled; flock + pgrep mentioned in context but not specified per-dispatch or global. Nothing prevents multiple runners spawning for same dispatch_id (e.g., via multiple `start-task` requests) or overlapping across dispatch_ids exceeding "max 2 concurrent".
  - **Why**: Race conditions on shared files (outbox overwrites), budget double-counting, vault corruption from parallel writes. Breaks completeness for concurrent dispatches and security (DoS via flooding).
  - **Fix**: Use per-dispatch flock file (`_openclaw/locks/{dispatch_id}.lock`) acquired on dispatch start. Global concurrency: pgrep limit + in-memory active set (or Redis). Reject new dispatch if >2 active or same dispatch_id running.

- **F4**
  - **Severity**: SIGNIFICANT
  - **Finding**: Budget enforcement has bypass via crash/recovery: In-memory ledger lost on runner crash; if stage completes post-crash, next runner sees stage output but no consumed budget, potentially spawning unlimited stages until files accumulate and timeout.
  - **Why**: Undermines budget as security boundary (BT5 DoS, runaway execution). Partial failures not covered—could exceed hard caps invisibly.
  - **Fix**: Persist cumulative `budget_consumed` in state file (F2). On stage-complete, sum metrics from *all* `-stage-N.json` files as fallback verification before next spawn.

- **F5**
  - **Severity**: SIGNIFICANT
  - **Finding**: Governance verification gaps: Runner verifies post-stage `governance_check`, but doesn't re-verify *during* stage (Claude could ignore CLAUDE.md mid-session). No check that stage deliverables' hashes match transcript claims. Canary (last 64 bytes) vulnerable to targeted tampering without full hash alert.
  - **Why**: Compromised Claude session (BT3) could produce bad output despite runner checks. Stage-level doesn't catch intra-stage degradation. Breaks security model.

- **F6**

- **F7**

  - **Why**: Malicious stage poisons chain (prompt injection across stages). `claude --print` sessions isolated but context chain vulnerable. Breaks security and feasibility for long chains.

## Synthesis

### Consensus Findings

**1. State machine: `running → blocked` transition is impossible** (OAI-F1, GRK-F1)
Both flagged that stage output is only readable after the session exits, making the `running → blocked` path unreachable. The blocked state can only be entered from `stage-complete`. The state machine diagram and transition rules are internally inconsistent.

**2. No persistence of dispatch state — crash recovery absent** (OAI-F8, GEM-F3, GRK-F2, GRK-F4)
Three reviewers independently flagged that the in-memory budget ledger is lost on runner crash, creating zombie dispatches and budget bypass vectors. This undermines both operability (solo operator can't recover) and security (budget as security boundary fails on crash).

**3. Budget enforcement trusts self-reported metrics** (OAI-F3, DS-F5)
Two reviewers flagged that tool calls, tokens, and wall time reported in stage output `metrics` are authored by the untrusted worker. A compromised session can underreport. Runner-observed measurements (wall time, stage count) should be the enforcement layer.

**4. Concurrency model underspecified** (OAI-F9, DS-F2, GRK-F3)
Three reviewers flagged ambiguity between "max 2 concurrent" (§9.4 BT5 row), flock-based single lock, and the DISPATCH_CONFLICT error code. The protocol doesn't specify whether dispatches queue, reject, or run concurrently.

**5. No mid-dispatch cancellation** (OAI-F10, GEM-F2, DS-F7)
Three reviewers flagged that users can't cancel a specific running dispatch without the global kill-switch. The CANCEL command exists in the escalation UI but has no lifecycle backend.

**6. Escalation option manipulation** (OAI-F6, OAI-F7, GEM-F5, DS-F4, GRK-F6)
All four reviewers flagged that structured-only escalation doesn't prevent manipulative option text crafted by a compromised or confused Crumb session. Options are worker-authored strings that get passed back into the resume prompt (worker→worker channel).

**7. Inter-stage prompt injection via `next_stage.instructions`** (GEM-F1, OAI-F5, GRK-F7)
Three reviewers flagged that `next_stage.instructions` are worker-authored content inserted into the next stage's prompt as if they were trusted instructions. A poisoned stage can propagate malicious instructions through the stage chain.

**8. Operability / debugging aids missing** (OAI-F17, DS-F9, GRK-F10)
Three reviewers flagged that solo-operator debugging is difficult without a centralized dispatch manifest or post-mortem artifact.

### Unique Findings

**OAI-F2: "Discard deliverables" on governance failure is unenforceable.** Genuine insight — vault writes happen during stage execution and cannot be undone by the runner. The protocol overpromises.

**OAI-F11: Inter-stage summaries insufficient; need structured handoff.** Genuine insight — summaries lose precision on filenames, decisions, and partial results. A bounded JSON handoff object fills the gap.

**OAI-F12: Status update overwrite loses history.** Minor operational concern — transcripts and stage outputs provide the real history. Status is a convenience channel.

**OAI-F14: Audit hash chaining underspecified.** Genuine insight — the concept is strong (all 4 praised it) but the spec lacks deterministic format (ordering, encoding, separator).

**OAI-F20: Claude Code may not write valid JSON reliably.** Genuine feasibility concern — runner needs retry/recovery for malformed stage output.

**DS-F12: Resource isolation between stages incomplete (shared filesystem).** Valid concern, but full sandboxing (temp workspaces, validated copy-in/copy-out) is disproportionate for Phase 2 with a single operator. Deferred to Phase 3.

**DS-F15: No rate limiting on escalation responses.** Low-risk — runner validates escalation_id matching, so duplicate responses for resolved escalations are naturally rejected.

### Contradictions

**None found.** All four reviewers broadly agree on the architectural strengths (role separation, stage isolation, budget-as-security, audit hash) and weaknesses (state machine correctness, crash recovery, self-reported metrics, escalation manipulation, inter-stage injection). Differences are in severity classification, not direction.

### Action Items

**Must-fix:**

**A1** — Fix state machine transitions. `running → blocked` → `stage-complete → blocked`. Add `running → failed` for session crashes. Add `stage-complete → canceled` for user cancellation.
*Sources: OAI-F1, GRK-F1, DS-F1*

**A2** — Persist dispatch state to disk. `_openclaw/dispatch/{dispatch_id}-state.json` with lifecycle_state, budget_consumed (runner-measured), pending escalation, stage counter. Atomic writes. Runner startup scan marks orphaned dispatches as `failed (RUNNER_RESTART)`.
*Sources: OAI-F8, GEM-F3, GRK-F2, GRK-F4*

**A3** — Make budget enforcement runner-observed. Wall time: runner-measured timestamps. Stage count: runner-counted. Tool calls and tokens: worker-reported metrics are advisory only for reporting; runner enforces hard wall-time cap and hard stage cap as the true security boundary. Remove token count as an enforcement dimension (keep for cost reporting).
*Sources: OAI-F3, DS-F5*

**A4** — Mitigate inter-stage prompt injection. Frame previous stage output as "Context from previous stage (UNTRUSTED)" not "Instructions." Inject static safety directives via `--append-system-prompt` on every stage. Add runner-side length cap and path validation on `next_stage.instructions` and `context_files` (must be within vault root, no `~/.ssh` etc.).
*Sources: GEM-F1, OAI-F5, OAI-F18, GRK-F7*

**A5** — Add `cancel-dispatch` operation and `canceled` terminal state. Runner checks for cancellation between stages (alongside kill-switch check). Cannot interrupt mid-stage — takes effect at next stage boundary.
*Sources: OAI-F10, GEM-F2, DS-F7*

**A6** — Clarify concurrency model: single active dispatch at a time (global flock). Remove "max 2 concurrent" language. New dispatches while one is active receive DISPATCH_CONFLICT and Telegram notification. Simplest model for solo operator.
*Sources: OAI-F9, DS-F2, GRK-F3*

**A8** — Change "discard ALL deliverables" on governance failure to "halt dispatch, write manifest of modified files to dispatch state, alert via Telegram." No rollback promise — partial work persists, user investigates manually or reverts via git.

**A9** — Add structured stage handoff object to stage output schema. `handoff: {}` — bounded JSON (≤8KB) with decisions, key facts, file pointers. Runner includes verbatim in next stage prompt alongside (not replacing) summaries. Large context → vault file via `context_files`.
*Sources: OAI-F11, DS-F6, GEM-F4*

*Sources: OAI-F14*

*Sources: OAI-F17, DS-F9, GRK-F10*

*Sources: OAI-F20*

**A13** — Align status values: stage output uses `done`/`next`/`blocked`/`failed`, final response uses `completed`/`error`. Add mapping note: `done` → `completed`, `failed` → `error`. These are different schema contexts (stage vs dispatch), not a bug, but document the mapping explicitly.
*Sources: GRK-F9*

### Considered and Declined

**DS-F3** (TOCTOU in governance hash computation): The window between hash computation and session spawn is single-digit milliseconds on a single-user system with no concurrent writers to CLAUDE.md. Post-stage verification catches any mismatch regardless. File locking adds complexity for negligible risk reduction. **Reason: overkill**

**DS-F15** (Rate limiting on escalation responses): Runner validates that `escalation_id` matches a pending escalation for a blocked dispatch. Duplicate responses for already-resolved escalations are rejected by this check. No additional rate limiting needed. **Reason: constraint** (already handled by escalation_id matching)

**GRK-F5** (Intra-stage governance degradation): Stage-level verification catches issues at boundaries. Mid-session governance monitoring would require modifying Claude Code internals, which is out of scope. The canary + hash check is the best available external verification. The `deliverables_hash` suggestion adds verification surface but still trusts the worker to compute it honestly — same trust model issue. **Reason: constraint** (not implementable without Claude Code modifications)

**DS-F13** (Telegram truncation for long summaries): Stage summary ≤500 chars plus status template chrome (~300 chars) totals ~800 chars — well within Telegram's 4096 char limit. The base bridge schema already handles message sizing. **Reason: incorrect** (arithmetic doesn't support the concern)
