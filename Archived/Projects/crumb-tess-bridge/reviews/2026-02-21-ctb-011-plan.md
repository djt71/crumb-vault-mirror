## Synthesis

### Consensus Findings

**1. Interactive wrapper must HOLD the lock, not just check it (4/4)**
OAI-F5 (CRITICAL), DS-F1, GEM-F3, implied by GRK-F2. The wrapper's "check then proceed" pattern is a textbook TOCTOU. The watcher can grab the lock between check and exec. Fix: wrapper acquires LOCK_EX, then exec's claude (fd stays open via exec), releases on process exit.

**2. Outbox existence check missing from dispatch pipeline (4/4)**
GRK-F1 (CRITICAL), GEM-F5 (CRITICAL), OAI-F8, DS-F8. The plan identifies this risk in the "Known Risks" section but doesn't include it in the pipeline flow. The crash-reprocess→DUPLICATE_REQUEST→overwrite scenario is a data loss bug.

**3. pgrep implementation needs specific filtering (4/4)**
OAI-F4, DS-F3, GEM-F4, GRK-F2. All four agree pgrep is brittle. Consensus: treat it as advisory, lock as authoritative. Filter out watcher PID + children + --print invocations. Make toggleable.

**4. Rate limiter seeding source of truth (3/4)**
OAI-F7, DS-F6, GRK-F4. Seeding from .processed/ but compacting .processed-ids is inconsistent. Must validate UUIDv7 format and handle malformed entries gracefully.

**5. Don't parse JSON in Python for reject path (3/4)**
OAI-F2, GEM-F2, DS-F16. Watcher doesn't need to parse request content. Pass filename to `bridge-processor.js reject`; Node handles ID extraction with existing robust parsing.

### Unique Findings

**Subprocess timeout (GRK-F3):** Only Grok flagged this, but it's clearly correct — a hanging node process blocks the flock indefinitely, creating a DoS. Genuine insight.

**Kill-switch permissions (DS-F4):** Only DeepSeek noted the kill-switch file is group-writable, meaning any crumbvault member (including openclaw) could disable the bridge. Valid security concern — kill-switch should be owner-only (tess).

**Umask in plist (GEM-F1, OAI-F12):** Two reviewers flagged missing umask. macOS LaunchAgent defaults may create files without group-write, breaking outbox consumption by openclaw. Valid operational concern.

**Kill-switch reject semantics (OAI-F6):** Should kill-switch reject consume messages (move to .processed/) or pause (leave in inbox)? Only OpenAI raised this. Valid design question.

**Log rotation (GRK-F7):** Only Grok. Valid but operational concern, not code concern.

### Contradictions

**Kill-switch semantics:** OAI-F6 says keep in inbox (pause), while the plan moves to .processed/ (consume). Both are defensible — consuming prevents replay after re-enable, pausing allows retry. This is a design decision for user approval.

**Lockfile stale cleanup:** DS-F11 says add stale lockfile cleanup on startup. But fcntl.flock() releases automatically when the owning process dies — this is a non-issue. DeepSeek may be conflating advisory locks with lockfile existence.

### Action Items

**Must-fix:**

- **A1** (OAI-F5, DS-F1, GEM-F3): Interactive wrapper must HOLD the lock during claude session. Acquire LOCK_EX on .bridge.lock, exec claude (fd inherited), release on exit. If lock held by watcher, warn and prompt.

- **A2** (GRK-F1, GEM-F5, OAI-F8, DS-F8): Add outbox existence check to dispatch pipeline. Before invoking processor, check if `{request_id}-response.json` exists in outbox. If yes, move request to .processed/ and skip. This requires extracting request ID from filename (files are named `{uuid}.json`).

- **A3** (GRK-F3): Add subprocess timeout (configurable, default 300s) to node bridge-processor.js invocation. On timeout: kill process, release flock, log TIMEOUT error, leave file in inbox for retry.

**Should-fix:**

- **A4** (OAI-F4, DS-F3, GEM-F4, GRK-F2): pgrep implementation: filter by full command line, exclude watcher PID + children + `--print` invocations. Make check toggleable via `CRUMB_BRIDGE_SKIP_PGREP` env var. Document as advisory (lock is authoritative).

- **A5** (OAI-F7, DS-F6, GRK-F4): Rate limiter: seed from `.processed-ids` file (parse UUIDv7 timestamps), not `.processed/` directory. Validate UUIDv7 format strictly, skip malformed entries with warning. Handle missing/corrupted file gracefully (start with empty window).

- **A6** (OAI-F2, GEM-F2, DS-F16): Reject subcommand: watcher passes filename only, not parsed JSON. Node's `rejectRequest()` extracts ID from file content (or falls back to filename). No JSON parsing in Python watcher.

- **A7** (OAI-F14, DS-F10): AC5 test: add integration test with `CRUMB_BRIDGE_USE_CLAUDE=1` using a stub `claude` script that validates invocation args/env and writes mock output. Assert watcher captures output correctly.

- **A8** (GEM-F1, OAI-F12): Add `Umask` key to plist (value 2 = 0002). In validation script, verify group-write on outbox files, inbox/.processed/, logs/. Document group permission requirements in runbook.

- **A9** (OAI-F10, DS-F9): File stability: watcher already ignores dotfiles and .tmp. Add documentation that Tess must use atomic write (write .tmp then rename) — this is already specified in bridge-schema.md §7 but should be referenced in watcher code comments.

**Defer:**

- **A10** (OAI-F6): Kill-switch reject semantics (pause vs consume). Evaluate during Phase 2 design. Current behavior (consume + error response) gives user feedback; pausing would leave them waiting.

- **A11** (GRK-F7): Log rotation. Handle operationally via newsyslog.conf or add RotatingFileHandler in Phase 2 hardening.

- **A12** (GEM-F6): Consecutive failure quarantine (.failed/ directory). Processor already handles errors; quarantine adds complexity. Add in Phase 2 if needed.

- **A13** (DS-F18): SIGHUP config reload. Nice-to-have; watcher restart via launchd is fast enough for config changes.

### Considered and Declined

- **DS-F2** (lockfile location atomicity): `incorrect` — fcntl.flock() is advisory by design; both participants cooperate. tess has write permission on _openclaw/ via group membership. Not a real issue.

- **DS-F11** (stale lockfile cleanup): `incorrect` — fcntl.flock() releases automatically when the owning process dies. No stale lock scenario with BSD flock semantics.

- **GRK-F9** (kill-switch content validation): `overkill` — kill-switch is an operator-only sentinel file. Adding content validation to a presence check is unnecessary complexity.

- **OAI-F17** (NFS filesystem check): `constraint` — vault is always local APFS per deployment requirements. Adding filesystem detection is out of scope.

- **DS-F4** (kill-switch group-writable): `constraint` — _openclaw/ is group-writable by design (shared between tess and openclaw). The kill-switch file inherits parent permissions. In practice, only tess creates it manually. If this becomes a concern, restrict via ACL in Phase 2 hardening.

- **OAI-F11** (backoff on interactive skip): `overkill` — the 60s fallback scan + 500ms kqueue timeout already rate-limits rescanning. The watcher won't busy-loop; it waits for the next kqueue event or fallback interval. Adding exponential backoff to a kqueue-driven loop is unnecessary.
