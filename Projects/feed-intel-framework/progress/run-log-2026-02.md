### Design Decisions
1. **Injectable API deps:** X adapter takes `XApiDeps` (pullBookmarks, searchTopics) as constructor arg. Keeps adapter testable without live credentials. Production wiring passes real HTTP clients.
2. **Preamble as standalone file:** Full X-specific preamble in `adapters/preambles/x.md` rather than inline YAML. Manifest carries a short summary in `prompt_preamble`; the adapter loads the full file at triage time.
3. **Thread detection ported verbatim:** Same 4-signal detection (structural, X API reply, TwitterAPI.io reply, text heuristic) with identical needs_context semantics. No behavioral change during migration.
4. **Exit code 3 for migration block:** Distinct from exit code 2 (pipeline lock) and 1 (runtime error). Allows launchd/monitoring to distinguish migration hold from operational failures.

### Compound Evaluation
- **Pattern: injectable dependency factory for adapters.** `createXAdapter(deps)` pattern will be reused for RSS/YouTube/HN adapters. Each adapter gets a factory that accepts source-specific API functions, returns a standard `CaptureAdapter`. This is the adapter isolation mechanism the spec describes in §6.
- **Convention: preamble_file in manifest, full text in adapters/preambles/.** Inline YAML preamble for validation, separate markdown file for the actual prompt. Avoids YAML multiline escaping issues and makes preambles reviewable as standalone prose.
- Routing: both patterns are implementation conventions, not yet stable enough for `docs/solutions/`. Monitor through RSS adapter (M3) before extracting.

### Design Decisions
1. **topic_weights + digest_messages table rebuilds:** SQLite doesn't support ALTER TABLE to change PKs. Used CREATE→INSERT→DROP→RENAME pattern within the transaction. Both tables had small row counts (0 and 7) so no performance concern. §8.1 step 2 says "Add source_type to ... topic_weights" but the composite PK requires a full rebuild — implementation is correct even though it diverges from the spec's literal wording.
2. **url_hash column:** Added even though §8.1 doesn't explicitly list it — the framework schema requires it and framework code queries it. Nullable with NULL default for all legacy rows; the X adapter normalizer computes it for new captures.
3. **Foreign keys OFF during migration:** Disabled during Stage 1 transaction to avoid constraint violations during canonical_id rewrite. Re-enabled after.
4. **Shared discovery cursor divergence:** Per §8.1 safety note, all 4 topic discovery streams start from the same shared lastScanRun. Each will diverge naturally over time. May cause some re-fetched (deduplicated) items but no data loss.
5. **Stage 2 fine-grained idempotency:** Uses INSERT OR IGNORE per row (checking actual `changes` count) rather than coarse-grained count(*) early-exit. If Stage 2 partially completes (e.g., curated written but discovery failed mid-way), a re-run fills in the missing entries without overwriting existing ones.
6. **Legacy digest_item_map table:** Survives migration untouched (442 rows). The framework never reads it — item maps are in adapter_state. Cosmetic cleanup candidate for FIF-025 (orchestrator) post-migration housekeeping.
