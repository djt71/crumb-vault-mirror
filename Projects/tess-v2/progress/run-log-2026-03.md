## 2026-03-28b — Conversation Analysis + Design Input

### Context
Danny shared a conversation thread with Opus (2026-03-27) covering eval-based vs. spec-based building, contract patterns, Ralph loops, agents as services, and overlay injection. Crumb analyzed for tess-v2 relevance.

### Key Finding: Spec + Eval Integration
Crumb's spec-based workflow and eval-based building are complementary, not competing. Specs define *how* to build. Evals (contracts) define *when done* and *whether good*. The integration: contracts derived from specs provide mechanical stop conditions, enabling autonomous execution without human approval at every step. This is the mechanism that moves orchestration from Crumb (interactive, Danny-driven) to Tess (autonomous, contract-driven).

### Five Patterns Extracted for Specification
1. **Contract-based execution** — mechanical stop conditions derived from action plans (spec+eval integration)
2. **Ralph loops** — focused iteration per contract, fresh context, accumulated failure feedback (spec+eval integration)
3. **Overlay injection at dispatch** — Tess passes behavioral overlays alongside contracts to executors
4. **Service formalization** — existing Tess functions classified as named services with defined contracts/interfaces
5. **Sycophancy-aware task framing** — neutral prompts for investigation, adversarial patterns for evaluation (design consideration, not committed)

### Architectural Decisions
- **AD-003:** Tess v2 integrates spec-based and eval-based building. Specs prescribe the work; contracts (derived from specs) provide mechanical completion verification. Phase transition gates are eval checkpoints.
- **AD-004:** Each dispatched task runs as a Ralph loop — one contract per session, fresh context, hard stop on contract satisfaction. No long-running multi-task sessions.
