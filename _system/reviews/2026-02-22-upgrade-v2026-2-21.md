# Peer Review: OpenClaw Upgrade Runbook v2026.2.17 → v2026.2.21

**Artifact:** `Projects/openclaw-colocation/design/upgrade-v2026-2-21.md`
**Mode:** full
**Reviewed:** 2026-02-22
**Reviewers:** GPT-5.2 (OpenAI), DeepSeek V3.2-Thinking, Grok 4.1 Fast Reasoning (xAI), Gemini 3 Pro Preview (Google)
**Review prompt:** Structured review of upgrade runbook evaluating correctness, completeness, security, operability, and rollback soundness against known deployment facts (dedicated user, LaunchDaemon, kill-switch patterns, filesystem IPC bridge).

---

## OpenAI (GPT-5.2)

- [F1]
- [Severity]: SIGNIFICANT
- [Finding]: Runbook claims "no required config changes," but later includes required/likely config interventions (Telegram `streaming` boolean migration; possible `agents.defaults.maxSpawnDepth` pin; doctor may change config).
- [Why]: Operators may skip careful config validation because the summary downplays changes; this increases risk of silent behavior drift (streaming, cost, agent spawning).
- [Fix]: Update Summary to "no *mandatory* changes expected, but verify/adjust: Telegram streaming key migration; consider pinning `maxSpawnDepth`; review doctor diff."

- [F2]
- [Severity]: MINOR
- [Finding]: Pre-check "Confirm gateway is running" uses `lsof -iTCP:18789 -sTCP:LISTEN` without asserting loopback binding.
- [Why]: Port could be listening on `0.0.0.0` (regression/misconfig) and the check would still pass; loopback-only binding is a key security control.
- [Fix]: Replace with a stricter check, e.g. `lsof -nP -iTCP:18789 -sTCP:LISTEN | grep -E "127\.0\.0\.1|localhost"` and fail if it shows `*:` / `0.0.0.0:`.

- [F3]
- [Severity]: SIGNIFICANT
- [Finding]: Backup of config is done as `sudo cp` into `/Users/openclaw/.openclaw/…` but without ensuring ownership/permissions remain correct.
- [Why]: Root-created files can change ownership/mode, potentially breaking the daemon or weakening hardening assumptions (e.g., group/world readability).
- [Fix]: Perform backups/restores as the `openclaw` user (or immediately `chown openclaw:staff` + `chmod` to the known-good mode). Example: `sudo -u openclaw cp ...` (or `install -m 600 -o openclaw -g staff ...`).

- [F4]
- [Severity]: SIGNIFICANT
- [Finding]: Stop procedure uses `bootout system/ai.openclaw.gateway` but does not validate the label actually matches the plist's `Label`, and does not fall back to bootout by path.
- [Why]: If the label differs (common in drift situations), `bootout` may no-op while returning non-obvious errors; the subsequent `pkill` might mask that LaunchDaemon state is still loaded.
- [Fix]: Add a validation step: `plutil -p /Library/LaunchDaemons/ai.openclaw.gateway.plist | grep Label` and/or `sudo launchctl print system/ai.openclaw.gateway`. If label mismatch, use `sudo launchctl bootout system /Library/LaunchDaemons/ai.openclaw.gateway.plist`.

- [F5]
- [Severity]: MINOR
- [Finding]: `pgrep -f "openclaw/dist/index.js"` is used as the process signature.
- [Why]: Upstream packaging can change entrypoints; a different node command line could evade detection.
- [Fix]: Also check by uid + port: `lsof -nP -iTCP:18789 -sTCP:LISTEN`; prefer port-based confirmation as source of truth.

- [F6]
- [Severity]: SIGNIFICANT
- [Finding]: Global npm install is done with cache export but not with explicit `npm_config_prefix` (yet PATH includes `/Users/openclaw/.local/bin`).
- [Why]: Without the prefix, npm may install to the system global prefix (`/opt/homebrew`) which the `openclaw` user can't write to.
- [Fix]: Add `export npm_config_prefix="/Users/openclaw/.local"` consistent with OC-008 baseline.

- [F7]
- [Severity]: SIGNIFICANT
- [Finding]: Restart uses `launchctl bootstrap` without first ensuring the job isn't already loaded.
- [Why]: Repeated `bootstrap` can error; operators may ignore and think the new version is running.
- [Fix]: Add bootout-before-bootstrap guard and explicitly check exit codes.

- [F8]
- [Severity]: MINOR
- [Finding]: No step to check LaunchDaemon logs after start.
- [Why]: Port listening and `openclaw status` may not surface early warnings.
- [Fix]: Add `log show --last 5m --predicate 'process == "node"'` or check LastExitStatus.

- [F9]
- [Severity]: SIGNIFICANT
- [Finding]: Doctor `--fix` auto-applies changes but runbook doesn't define an allowlist of acceptable new keys for this upgrade.
- [Why]: "No unexpected new keys" is too strict for a version upgrade that may legitimately add keys.
- [Fix]: Enumerate acceptable new/changed keys for this upgrade; treat anything outside the list as NO-GO.

- [F10]
- [Severity]: CRITICAL
- [Finding]: V2 (Gateway Password) verification says "connect to gateway and authenticate" but provides no safe method.
- [Why]: Password handling can accidentally persist secrets in terminal history, logs, or screenshots.
- [Fix]: Document a safe verification method that reads password from prompt without echo, or reference an `openclaw auth test` command.

- [F11]
- [Severity]: SIGNIFICANT
- [Finding]: Telegram streaming guidance suggests setting `channels.telegram.streaming: true` but doesn't state current deployed key or how to detect it.
- [Why]: Operator may add a duplicate/incorrect key path.
- [Fix]: Add "inspect current telegram config block" command and precise JSON path.

- [F12]
- [Severity]: MINOR
- [Finding]: Isolation tests run twice (once for gate, once for saving), potentially with different outcomes.
- [Why]: Double-run wastes time and can create confusion if flaky.
- [Fix]: Run once, tee output to file.

- [F13]
- [Severity]: SIGNIFICANT
- [Finding]: SBOM snapshot doesn't capture `npm -v`, `node -v`, or global prefix path.
- [Why]: Runtime/toolchain drift is a common upgrade regression source.
- [Fix]: Add `node -v`, `npm -v`, `npm prefix -g`, `which openclaw` under the openclaw user.

- [F14]
- [Severity]: SIGNIFICANT
- [Finding]: No explicit verification that LaunchDaemon still runs as user `openclaw` (not root) after restart.
- [Why]: Running as root breaks OS-level isolation — a major security regression.
- [Fix]: `ps -o user,pid,command -p $(pgrep -f ...)` or `lsof -iTCP:18789 -sTCP:LISTEN` and confirm owner is `openclaw`.

- [F15]
- [Severity]: MINOR
- [Finding]: Bridge verification uses `~/crumb-vault/_openclaw/inbox/` — ambiguous `~` (tess or openclaw?).
- [Why]: Ambiguity can cause operator to test wrong directory.
- [Fix]: Use absolute path `/Users/tess/crumb-vault/_openclaw/inbox`.

- [F16]
- [Severity]: MINOR
- [Finding]: Rollback reinstalls `openclaw@2026.2.17` but npm may resolve transitive dependencies differently.
- [Why]: Rollback may not be a true rollback if deps shifted.
- [Fix]: Archive pre-upgrade `npm pack` or validate post-rollback `npm ls` against snapshot.

- [F17]
- [Severity]: SIGNIFICANT
- [Finding]: No explicit kill-switch linkage during upgrade.
- [Why]: Upgrades are when misconfig is most likely; kill-switch runbook exists but isn't referenced.
- [Fix]: Add conditional: "If port binds non-loopback or tailscale toggles on, execute kill-switch immediately" with runbook path.

- [F18]
- [Severity]: STRENGTH
- [Finding]: Correctly aligns with key deployment facts: dedicated user, LaunchDaemon, bootout+pkill, bootstrap, HOME/PATH exports, 9-test gate, Telegram awareness, bridge filesystem IPC.

- [F19]
- [Severity]: STRENGTH
- [Finding]: Config diff review explicitly checks security-relevant settings and treats hardening removal as NO-GO.

- [F20]
- [Severity]: MINOR
- [Finding]: Some `lsof` commands omit `-nP`; some places lack expected output examples.
- [Fix]: Use `lsof -nP ...` and add concrete expected-line examples.

## Synthesis

### Consensus Findings

**CF1: Missing `npm_config_prefix` — install will likely fail (OAI-F6, GEM-F1)**
Two reviewers flagged that the `npm install -g` commands export `npm_config_cache` but not `npm_config_prefix`. Since `PATH` includes `/Users/openclaw/.local/bin`, the original OC-008 install clearly used a user-local prefix. Without it, npm attempts to install to `/opt/homebrew/lib` which the `openclaw` user can't write to. Gemini rated this CRITICAL (install fails outright); OpenAI rated SIGNIFICANT (drift risk). **Verdict: CRITICAL — this will break the upgrade.**

**CF2: pgrep/pkill not scoped to openclaw user (OAI-F5, DS-F2, GRK-F2)**
Three reviewers flagged that `pgrep -f "openclaw/dist/index.js"` could match processes from other users. All recommend adding `-u openclaw`. **Verdict: SIGNIFICANT — easy fix, reduces blast radius.**

**CF3: Config backup too narrow (OAI-F3, GRK-F3, GEM-F2)**
Three reviewers flagged that only `openclaw.json` is backed up. Grok and Gemini recommend backing up the entire `.openclaw/` directory to cover tokens, state, and potential schema migrations. OpenAI focused on ownership/permissions of the backup file itself. **Verdict: SIGNIFICANT — full directory backup is safer.**

**CF4: Gateway password verification (V2) is vague (OAI-F10, GRK-F4)**
Two reviewers rated this CRITICAL. The runbook says "connect to gateway and authenticate" without a command, endpoint, or safe password-handling method. **Verdict: SIGNIFICANT — should specify a method, but downgraded from CRITICAL because the gateway is loopback-only and this is a verification step, not a security boundary.**

**CF5: lsof check should verify user and loopback binding (OAI-F2, GRK-F5, OAI-F14)**
Multiple reviewers noted that `lsof -iTCP:18789` doesn't verify the listener is `openclaw` on `127.0.0.1`. A `0.0.0.0` bind or root-owned process would pass undetected. **Verdict: SIGNIFICANT — add `-nP` and grep for both `127.0.0.1` and user `openclaw`.**

**CF6: Telegram streaming config needs proactive verification (OAI-F11, GRK-F7)**
Two reviewers flagged that the runbook defers streaming verification to a manual test rather than proactively checking the config key. **Verdict: MINOR — the manual test covers this adequately; config check is a nice-to-have.**

**CF7: LaunchDaemon/plist compatibility check (DS-F4, GRK-F9)**
Two reviewers noted no check for plist or wrapper script compatibility with the new version. **Verdict: MINOR — npm upgrade doesn't touch the plist or wrapper script; this is a valid but unlikely edge case.**

### Unique Findings

**UF1: Data/state backup for schema migration rollback (GEM-F2)** — Gemini uniquely raised that if v2026.2.21 migrates a database schema, restoring the old binary with the new schema would crash. This is a genuine insight for stateful applications. Whether OpenClaw has a vector store or database that could be migrated is uncertain, but defensive backup of the full `.openclaw/` directory costs little. **Genuine insight.**

**UF2: Kill-switch linkage during upgrade (OAI-F17)** — OpenAI uniquely noted that upgrades are peak-risk for misconfig, and the runbook should reference the kill-switch runbook with explicit trigger conditions (non-loopback bind, tailscale on). **Genuine insight — low cost to add.**

**UF3: npm install error handling (GRK-F1)** — Grok uniquely flagged that `bash -c` may swallow non-zero exit codes. A post-install version check acts as a catch. **Genuine insight.**

**UF4: Doctor expected-changes allowlist (OAI-F9)** — OpenAI suggested enumerating acceptable new keys per upgrade rather than a blanket "no unexpected keys." **Reasonable for structured upgrades but over-engineering for a point release — defer.**

**UF5: SBOM should capture node/npm versions (OAI-F13)** — Only OpenAI flagged this. Runtime version capture alongside package SBOM is good practice. **Genuine insight — easy to add.**

**UF6: Process owner verification post-restart (OAI-F14)** — Only OpenAI flagged verifying the gateway runs as `openclaw` not `root` after restart. Running as root would catastrophically break the isolation model. **Genuine insight — critical security check.**

**UF7: Isolation test double-run (OAI-F12)** — Use `tee` to run once and save output simultaneously. **Genuine insight — minor but obvious fix.**

### Contradictions

**C1: npm_config_prefix — CRITICAL vs SIGNIFICANT**
Gemini rated the missing prefix as CRITICAL (install will fail with EACCES). OpenAI rated it SIGNIFICANT (drift risk). The actual severity depends on whether the original install used `--prefix` or a global npm config. Given PATH includes `.local/bin`, Gemini's reading is more likely correct — the install will fail outright. **Resolved: CRITICAL.**

**C2: Summary accuracy**
OpenAI flagged the "no required config changes" summary as misleading (F1). DeepSeek and Grok didn't flag this — their reviews implicitly accepted the framing. The summary is technically accurate (nothing is *required* before update) but could be clearer about post-update verification. **Resolved: MINOR wording fix.**
