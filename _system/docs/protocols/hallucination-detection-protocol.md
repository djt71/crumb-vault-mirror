# Hallucination Detection Protocol

Extracted from design spec §4.8. This is the authoritative reference — the spec contains a summary.

**Apply when:** Non-code outputs, compound insights, pattern extraction, context retrieval, summary generation.

**Purpose:** Prevent LLM hallucinations from corrupting the system's knowledge base or influencing decisions.

**Activation tiers:** Not every check fires on every operation. Checks are tiered by weight and activation frequency:

| Tier | Checks | When |
|---|---|---|
| **Always-on** | Confidence tagging, context relevance justification, falsifiability check (compound step only), interpretation flagging | Every relevant operation — lightweight, embedded in workflow |
| **Risk-proportional** | Provenance check | When consuming a summary not generated in the current session (see §2) |
| **Audit-time** | Calibration tracking review, deep provenance analysis, summary spot-checks | Weekly audit or when failures are discovered |
| **Human-grounded** | User validates one pattern + one summary against their own knowledge | Monthly audit (§7) — the only tier that introduces external verification |

## 1. Falsifiability Check (Always-on — compound step only)

**For patterns and insights:**
- Can I name a counterexample where this doesn't apply?
- If no counterexamples exist, the claim is too broad (likely hallucinated or overgeneralized)

**Example:**
- ❌ "Always use microservices for scalability" → No falsifiability, too broad
- ✓ "Use microservices when services have independent scaling needs and team autonomy exists" → Can identify cases where this doesn't apply

## 2. Provenance Check (Risk-proportional)

Provenance checking guards against summary drift — but drift risk varies by context. Apply checks proportional to the gap between the summary and its source:

**When loading a summary generated by a subagent (agent gap):**
- Verify key constraints and decisions match the subagent's full output
- Check that no interpretive claims were introduced that aren't in the source
- This check is part of the existing subagent validation step (spec §3.2.1, §3.2.2)

**When loading a summary from a previous session (time/context gap):**
- Verify summary freshness mechanically: compare summary's `source_updated` against parent's `updated` field. If mismatched, regenerate before proceeding.
- Spot-check one critical claim against the parent document (pick the highest-stakes constraint or decision)
- If anything looks off, regenerate the summary before proceeding

**When loading a summary generated in the current session (no gap):**
- Skip provenance check — the source is in context, drift risk is near-zero

**For patterns (always-on, lightweight):**
- Which specific tasks generated this insight?
- List the task IDs or project names that support this pattern
- If the pattern came from a single instance, flag as `tentative-pattern`

**For context (always-on, lightweight):**
- Why is this document relevant to the current task?
- State relevance explicitly: "Loading [filename] because [specific reason tied to current task]"
- If relevance is unclear after reading, discard the document

**Deep provenance analysis (audit-time only):**
During the weekly audit summary spot-check, apply the full provenance analysis to 2-3 summaries:
- Are key claims quoted verbatim from source or paraphrased?
- If paraphrased, can you point to the source sentence/section?
- Are there any interpretive claims not explicitly flagged with `[INTERPRETATION]`?
- Log any drift found to `_system/docs/failure-log.md`

## 3. Calibration Tracking (Audit-time — log on discovery, review during audits)

**Log to `_system/docs/failure-log.md` when:**
- Convergence says "adequate" but human rejects output
- Compound insight proves false in later projects
- Subagent validation passes but implementation fails
- Summary contains information not in source document
- Context retrieval loads irrelevant documents
- Routing sends a task to the wrong skill or workflow
- Output significantly misses the intended scope or quality bar
- Any session that went clearly poorly — the failure log captures the diagnosis autonomously

**Canonical failure type taxonomy:**

Every failure-log entry is classified as one of the following types. Hallucination-related incidents appear as specific types within this taxonomy (typically Bad Summary, Irrelevant Context, or Convergence Failure) — the broader taxonomy captures failure modes beyond hallucination.

- **Convergence Failure** — Convergence says "adequate" but human rejects output
- **False Pattern** — Compound insight proves false in later projects
- **Bad Summary** — Summary contains information not in source document
- **Irrelevant Context** — Context retrieval loads irrelevant documents
- **Routing Failure** — Task sent to wrong skill, workflow, or domain classification
- **Scope Miss** — Output significantly misses the intended scope
- **Quality Miss** — Output meets scope but falls short on quality bar
- **Wrong Skill** — Correct domain/workflow but wrong skill invoked for the task

**Log entry format:**
```markdown
## [Date] - [Type: Convergence Failure | False Pattern | Bad Summary | Irrelevant Context | Routing Failure | Scope Miss | Quality Miss | Wrong Skill]

**What happened:** [Description]
**Why it passed validation:** [What check missed this]
**Actual failure mode:** [How it manifested]
**System update:** [What changed to prevent recurrence]
```

**Use calibration data to:**
- Tighten convergence rubrics when failures recur
- Update validation procedures in skills
- Identify blind spots in quality assessment

## 4. Confidence Tagging (Always-on)

All patterns in `_system/docs/solutions/` must include confidence level in frontmatter:

```yaml
confidence: high  # Backed by tests, measurements, or 3+ empirical instances
# OR
confidence: medium  # Reasonable inference from 1-2 instances, needs validation
# OR
confidence: low  # Speculative, should be tagged `tentative-pattern`
```

**Confidence levels guide usage:**
- `high`: Can be promoted to skill rules or CLAUDE.md
- `medium`: Use with caution, validate in next 1-2 projects
- `low`: Reference only, do not build automation around

**Promotion rules:**
- `low` → `medium`: After validation in 1 additional project
- `medium` → `high`: After validation in 2 additional projects
- Any confidence level can drop if contradictory evidence emerges

## 5. Summary Generation Rules (Always-on — simplified)

When generating summaries, apply these lightweight rules as part of normal workflow:

**Preserve critical content verbatim:**
- Don't paraphrase critical constraints, requirements, or decisions
- Use quotes for exact language: `"The API must support pagination for lists >100 items"`

**Flag interpretations:**
- If the summary draws conclusions not explicit in source: mark as `[INTERPRETATION]`
- Example: `[INTERPRETATION: This suggests we need real-time updates]`

**Source-level citation (high-stakes documents only):**
- For specifications, architecture decisions, and API contracts: include section references (e.g., "See specification.md §3.2 for rationale")
- For routine summaries (progress updates, meeting notes, domain summaries): skip section-level citations — verbatim preservation and interpretation flagging are sufficient

**Drift detection (audit-time — via audit skill):**
- Audit skill spot-checks summaries weekly with deep provenance analysis (§2)
- If drift found despite preventive mechanisms: regenerate all summaries for that project, investigate whether the regenerate-on-modify step is being consistently followed
- Log summary drift to `_system/docs/failure-log.md`

## 6. Context Relevance Justification (Always-on)

**When loading a document from vault:**
- Explicitly state why it's relevant to current task
- Format: "Loading [filename] because [specific reason tied to current task]"
- If relevance is unclear after reading: discard document, don't use it in analysis

**Examples:**
- ✓ "Loading career-transition-pattern.md because current task involves career change to tech industry"
- ✗ "Loading career-transition-pattern.md because it might be useful"

**For searches returning multiple documents:**
- Rank by relevance to current task
- Load top 3-5 only
- Explain ranking: "Prioritizing X over Y because current task emphasizes [aspect]"

## 7. Human-Grounded Validation (Monthly — via audit skill)

The checks above are all self-referential — the same LLM evaluates its own outputs. This monthly check introduces an external grounding point the system cannot game.

**During the monthly audit, the audit skill selects 2 items for human review:**

1. **One `high`-confidence pattern** from `_system/docs/solutions/` — present the user with the pattern and its evidence chain (which tasks generated it, when confidence was promoted). Ask: "Does this match your experience? Is this actually a reliable pattern?"
2. **One summary of a high-stakes document** (spec, ADR, or API contract) — present the summary alongside 2-3 key claims extracted from it. Ask: "Do these claims match your understanding of the document?"

**If either fails validation:**
- Log the failure to `_system/docs/failure-log.md` with specifics
- Trigger a deep provenance analysis (§2) of all patterns/summaries in that category — not just the failed item
- If a high-confidence pattern fails, demote to `medium` and flag for re-validation

**Cost:** Two questions per month. Minimal human burden, but provides a calibration signal that no amount of self-evaluation can replicate.
